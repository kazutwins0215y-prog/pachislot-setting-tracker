## preprocess.py

### データの実態(確認済み制約)

| 制約 | 内容 |
|---|---|
| ART列は店舗によって使われ方が異なる | 多くの店舗ではART列は全行NULL(AT/CZ機の当りもBB/RB列に一括格納)だが、店舗によっては他店でBB/BB確率に入るデータがART/ART確率列に入っている(データ提供元のページ仕様が店舗ごとに異なるため。2026-07、エスパス日拓新宿歌舞伎町店で確認)。Stage1aの`resolve_rate_columns`が店舗ごとに実データで検証し自動吸収する |
| AT機の理論値不一致 | BB/RB確率が公開設定差表と一致しない機種あり(例: ヴァルヴレイヴ2のRBは逆相関) |
| 号機(実機固有ID)は取得不可 | 台同一性キー = (機種名, 台番号) |
| 欠損行の形式 | 識別子は実値・計測値NULL・判定不能フラグ列あり |

### Stage 0: データ正規化

- TEXT→数値変換(カンマ除去・±符号対応)
- 日付・台番号・機種名でユニーク化
- 1行 = 1台×1日のサンプル単位

### Stage 1a: 機種別Tierキャリブレーション(機種×データ提供元で共通)

BB/RB/ART列を3段階に分類し、信頼性に応じた使い方を決める。

| Tier | 条件 | 使い方 |
|---|---|---|
| A | 観測確率が公開設定差表と一致 | 理論値 $p_s$ をそのままベイズ尤度に使用 |
| B | 理論値未対応だが差枚率との相関 \|r\|>0.5 | 絶対確率では使わず、同機種内でパーセンタイル化して相対スコア化 |
| C | 理論値とも差枚率とも無関係 | 切り捨て、他指標で代替 |

**実証済み判定例**(回転数≥1500サンプル):

| 機種 | BB↔差枚率 | RB↔差枚率 | 判定 |
|---|---|---|---|
| マイジャグラーV | — | — | Tier A(BB=BIG, RB=REGそのまま) |
| 東京喰種 | r=0.96 | r=0.41 | Tier B、BB+RB合成で使用 |
| ヴァルヴレイヴ2 | r=0.95 | **r=-0.06(逆相関)** | BB=Tier B、**RB=Tier C(切り捨て)** |
| 戦国乙女4 | r=0.82 | r=0.89 | Tier B、合成確率(相関0.905)で使用 |

**店舗ごとの列マッピング自己検出**(`resolve_rate_columns`、2026-07追加):
- specsのTierラベル(BB/RB)は機種の性質を表すが、**どの物理列(BB確率/RB確率/ART確率)がそのチャンネルの実データを持つかは店舗ごとに異なりうる**(データ提供元サイトのページ仕様が店舗ごとに異なるため)
- `calibrate_all`は機種ごとに、BB確率/RB確率/ART確率の3列それぞれと差枚率の相関を**その店舗の実データで**計算し、最も強く相関する列(|r|>0.5、サンプル数30以上)をTier Bチャンネルの参照列として採用する(`column_map`)。同一列を'BB'・'RB'で重複採用はしない
- specsのラベルに対応する列(BB確率など)がこの店舗でたまたま欠損していても、実データが別列(ART確率など)にあれば自動的にそちらを参照する。どの列も相関が閾値未満ならそのチャンネルは寄与0として扱う(判定不能の代入はしない)
- `compute_all_logLR`にはこの`column_map`を渡し、Tier B分岐で決め打ちの列名の代わりに使用する

**152機種への一括Tier判定**(`fase2/assign_tier.py`、2026-07時点):
- ノーマルタイプ(BIG/REG方式、AT/CZ要素なし)→ 相関計算せず無条件でTier A(AT/CZによる理論値ズレ要因が無いため。マイジャグラーVと同じ扱い)
- それ以外(AT/CZ機)→ 上記と同じ相関ロジック(|r|>0.5)を全店舗データで実行しB/Cを自動判定
- 判定結果: BB Tier(A=11 / B=132 / C=9)、RB Tier(同上)
- 生データは`fase2/raw_specs_scraped.json`(chonborista.comのスクレイピング結果)に保持。列名→BB/RB/payoutの対応は各機種の`source_columns`で確認可能

**機種スペック自動取得・90日凍結ルール(2026-07-20導入)**:
- 新台の理論値が取得されない問題への対応として、`scrape_machine_specs.py`を`fase4/run_daily.py`へ
  自動組込(`assign_tier.py`も続けて自動実行。5日おき。呼び出し間隔自体はrun_daily側の
  `fase4/specs_refresh_state.json`が管理し、90日凍結ルール自体はscrape_machine_specs.py側が管理)
- 各機種の`raw_specs_scraped.json`エントリに`first_seen`(レプリカ初出日)・`last_attempt`(最終試行日)・
  `frozen`(true固定で以後再取得しない)を付与。**初出から90日以内は再取得対象、90日超過でfrozen確定**
  (未解決のまま凍結した場合はstatusを`gave_up`にする)
- **一度もscrape_one()を試みていない機種(last_attempt未記録)は、真の初出日がどれだけ過去でも
  必ず1回はスクレイプ対象にする**(凍結判定は課さない)。実装当初、この判定を`first_seen`キーの
  有無で行っていたところ、1回の実行件数上限(`MAX_MACHINES_PER_RUN`)に切り詰められて未処理のまま
  終わった機種が次回「もう初回ではない」と誤判定されgave_up凍結される事故が2026-07-20の実データ
  確認で発覚したため、判定基準を`last_attempt`の有無に修正した
- 既存機種(2026-07-20導入時点で153機種)は`migrate_specs_freeze.py`で一括移行済み: 取得済み
  (ok/needs_review)→即`frozen`、未取得(not_found/ambiguous/error)→移行日を`first_seen`とした
  新しい90日猶予。**この移行はraw_specs_scraped.jsonに既存エントリがある機種のみが対象**であり、
  レプリカにはあるが同ファイルに一度も記録がなかった機種(2026-0720時点で45件。手動実行が
  途絶えていた期間に新規登場していた)は上記の「未試行は必ず1回」ルールでそのまま自然に回収された
- 1回の実行の処理件数上限は`MAX_MACHINES_PER_RUN`(30、保険。通常の5日おき運用では数件のはず)
- 403は即中断・429はリトライ上限到達で中断(残り機種はスキップし取得済み分のみ保存。fase1と同様の
  サーキットブレーカー方式)。SSL検証は`truststore`方式に統一(fase1と同じNorton対策)
- JSON書き込みはアトミック化(一時ファイル→`os.replace`)。**OneDrive同期フォルダ上で動作するため
  同期エージェントによる一瞬のファイルロック(`PermissionError`/WinError 5)が実データ確認で
  複数回発生**しており、`os.replace`失敗時に短間隔で最大5回リトライする

**「ノーマルタイプ」判定はBIG/REG列名だけでは不十分(2026-07判明・修正済み)**:

- 修正: `scrape_machine_specs.py`がchonborista.comの「機種概要」テーブルの「仕様」欄
  (例: `ノーマル`/`ノーマルタイプ`/`AT機`/`AT機(スマスロ)`)を`machine_type`として追加取得し、
  `assign_tier.py`の`is_confirmed_normal_type`が「仕様」欄が`ノーマル`/`Aタイプ`で始まる
  場合のみTier A確定を許可するようにした(それ以外は列名が一致していても相関ベースの
  Tier B/C判定にフォールバックする)
- 修正後もキングハナハナ-30・マイジャグラーVの一部の行に残った異常(9件)は、Tier判定とは

**回転数の機械的バイアス補正**(AT中除外問題):
- AT機はAT中の回転数がデータに含まれないため、当りが多い台ほど記録上の回転数が実態より少なくなる
- 同一データ提供元なら店舗を跨いで再現する → 機種ごとに補正式を1回校正すれば全店舗で再利用可能

### Stage 1b: 店舗別行動成分(複数店舗データ前提、`multi_store.py`で実装済み)

- 繁盛度・客層効果による回転数チャンネルの判別力の強さは店舗ごとに異なる
- 単店データのみでは学習不可。Stage 5の階層モデルで複数店舗データから自動学習

**実装(2026-07、`multi_store.learn_bin_curves`)**:
- チャンネル③の入力`kaiten_zscore`(台内=同一ホール名×機種名×台番号の回転数zスコア、`preprocess.compute_kaiten_zscore`。履歴5日未満はNaN)を全店舗データでプールし、機種ごとにデシル分けして「そのデシルの平均logLR_rng(RNGベースの疑似正解y1) − 機種全体平均」を実測カーブとして学習する
- 標本数200件未満の機種はカーブを持たない(＝該当機種はStage2でlogLR_kaiten=0.0のまま)
- 学習結果は`kaiten_bin_curves.json`に保存され、Stage1a(Tier判定)と同じ「1回学習・全店舗再利用」方式で`preprocess.compute_all_logLR`が自動読み込みする(単店実行の`app_a.py`/`app_b.py`は無改修で恩恵を受ける)

**⚠ 循環学習対策(2026-07再設計)**: カーブの教師がy1(logLR_rng)であるため、素のカーブ値をStage3で`w1×y1`に加算するとRNG証拠の**二重計上**になる。このため現在は:
- **w3の既定値は0.0(チャンネル無効)**。Stage5のLOSO交差検証(`validate_kaiten_channel`)に合格した場合のみ正のw3が保存される
- 有効化時もカーブ値そのままではなく、**直交化残差**(カーブ値 − (orth_a + orth_b×y1)、y1で説明できる成分を除いた「回転数だけが持つ独立成分」)をチャンネル③として使う(`compute_logLR_kaiten_column`のorth_a/orth_b引数)

### Stage 2: 証拠チャンネルごとのlogLR計算

3チャンネルを**個別に**計算する。

| チャンネル | 内容 | 計算方法 |
|---|---|---|
| ① RNG確率 | Tier A/Bと判定された列 | ポアソン近似: $\ln L(s) = k\ln(np_s) - np_s$ |
| ② 差枚(出玉) | 分散補正済み | 正規近似 **per-game スケール**: $\text{logLR} = \frac{\bar{x} \cdot \mu_s - \mu_s^2/2}{\sigma^2}$（$\bar{x}=\text{差枚}/n$）、σは機種別校正 |
| ③ 回転数(行動) | Stage1a/1b補正済み | 台内zスコアをデシル分けし、実測logLR曲線(非線形・非対称)の**直交化残差**を使用。**既定は無効(w3=0.0)**で、Stage5のLOSO交差検証合格時のみ有効化 |

**チャンネル②のμ_s推定(2026-07更新)**:
- `machine_setting_specs.json` に理論機械割(payout)がある機種(2026-07時点で153機種)は、高設定側の設定の平均payoutから理論期待差枚/G($\mu_s = (\overline{\text{payout}_{\text{高設定}}} - 1) \times 3$、3=標準ベット枚数)を**そのまま**使用する
- **高設定/低設定の分割境界(2026-07-14修正)**: 設定は機種によらず1〜6の共通尺度(2なし・3なし等の欠番あり)のため、**設定4以上=高設定・設定3以下=低設定の固定境界**で分ける(`preprocess._split_setting_keys`。チャンネル①のp_s/p_baselineとチャンネル②のμ_sで共用)。旧実装は「設定リストの上位半分/下位半分」折半で、欠番パターンにより境界がズレていた(沖ドキ系(1,2,3,5,6)は設定3が高設定側に混入して基準が甘く、ピンクパンサーSP(1,4,5,6)は設定4が低設定側に混入して過剰に厳しかった。修正の影響は6機種のみ・他146機種は不変)
- 無い機種は従来通り実測差枚率の75パーセンタイルを経験的に使用(30サンプル以上必要)
- 店舗単位の一律補正(κ)は**入れていない**。実データ検証の結果、店舗全体の実測平均差枚が理論値より低いのは「低設定運用が多い」という設定配分の反映であり、個々の台が高設定だった場合の期待値のズレではないため。一律補正をかけると高設定検出力そのものを削ぐことが判明した(詳細: `preprocess.compute_all_logLR`のコメント参照)

回転数の**2つの役割**を区別する:
- 役割①: サンプルサイズ(回転数が多いほどチャンネル①の尤度がシャープになる) → チャンネル①の n に内包
- 役割②: 独立証拠(客が示唆を見て回すという行動の証拠) → チャンネル③として独立集計

> セクション7の稼働推定(店全体の回転数集計)とは目的が異なる点に注意。同じ「回転数」カラムから異なる意味の指標を導出している。

### Stage 3: 統合スコア化

$$\text{LogOdds}(\text{高設定}) = \beta_0 + w_1 \cdot \text{logLR}_1 + w_2 \cdot \text{logLR}_2 + w_3 \cdot \text{logLR}_3$$

- 出力: `sigmoid(LogOdds)` → 高設定確率(0〜1)。または LogOdds そのもの
- 設定3欠番など機種仕様の差にも自然に対応(1〜6の個別事後分布不要)
- 単店段階では $w_3$ は実測相関の強さをそのまま使用。最終的な重みはStage5で学習
- **実装(2026-07)**: `preprocess.compute_log_odds`はw1/w2/w3を明示指定しない場合、`stage3_channel_weights.json`(Stage5が学習・保存)を自動読み込みする。未学習時は既定値(w1=1.0, w2=0.5, w3=1.0)にフォールバックするため、既存の呼び出し側(`app_a.py`/`app_b.py`)は無改修のまま学習結果を反映できる

**事前確率切片β₀(2026-07-07導入)**:

- $\beta_0 = \ln(\pi/(1-\pi))$。$\pi$=「証拠ゼロの台日が高設定である確率」(店舗の高設定投入率の事前推定)で、`stage3_channel_weights.json`の`prior_high_ratio`(既定0.15=`preprocess.DEFAULT_PRIOR_HIGH_RATIO`、業界感覚に基づく暫定値)から読み込む
- 旧実装はβ₀なし=暗黙にπ=0.5で、証拠ゼロの行がhigh_prob=0.5になり、店舗集計E[高設定台数]/Nが0.5近辺の狭帯域に集中する原因だった(上限キャリブレーション過剰発動(発動日割合3.8%〜82.7%)の根本原因)
- β₀は全行一律のため**店舗内の台のランキング(log_oddsの順序)は不変**。変わるのはhigh_probの絶対水準(確率としてのキャリブレーション)で、店舗横断の集計(E[高設定台数]/N・上限モデル・S_全台系の絶対値)が意味を持つようになる
- あわせて`patterns._compute_event_scores`(S_新台増台/S_移動台)の基準値フォールバック0.5を廃止(基準値が計算不能な店舗×機種は「検出不可」として除外。β₀導入後は0.5が「中立」を意味しないため)
- `multi_store._save_channel_weights`はStage5の学習結果を保存する際、学習対象外の`prior_high_ratio`を既存ファイルから保全する(学習再実行で消えない)
- 将来: πの店舗別学習は上限モデルStep2(Kalman)と統合して検討([`今後の実装予定.md`](今後の実装予定.md)2.1節)

**チャンネル間スケール差の注意点（実装メモ）**

差枚チャンネル②は per-game 正規化済みだが、設定間の識別力は RNG チャンネル①より本質的に高い（実測で 5〜10 倍程度）。そのため暫定的に $w_2 = 0.5$ に抑えている。Stage5 の重み学習後に適切な値に更新すること。

また、チャンネル①は O(n_daily) — 1日の回転数に比例してわずかに増加 — なのに対し、チャンネル②は per-game 正規化で O(1) となる非対称性が残る。将来的に完全統一するなら①も n で割る選択肢がある。

> **このStage3の出力(LogOdds/高設定確率)が、patterns.pyのパターン検出・サブスコアへの入力データになる**

### Stage 4: 信頼度しきい値・判定保留

- 判定不能条件: 各指標の期待発生回数が目安5回未満(= 回転数が少なすぎる台日)
- 判定不能日は**低設定・中間値などの代入を行わず単純除外**する(虚構の値を作らない)

**RNGチャンネル異常値検出(`mark_rng_anomaly`、2026-07追加)**

データ提供元(ana-slo.com)側の表記ミスにより、Tier A機種のBB/RB確率が設定6(理論上の
最良ケース)を大幅に超える行がまれに存在する(実例: マルハン新宿東宝ビル店キングハナハナ-30
774番、回転数8922でBB=666=観測確率0.075。理論値は設定6でも0.0043で17倍の乖離)。

- 設定6理論確率を仮定した片側Poisson検定(`scipy.stats.poisson.sf`)で、観測回数が
  統計的に説明不能なほど過剰(p < `RNG_ANOMALY_ALPHA`=1e-6)な行を検出し、Stage4の
  判定不能と同じ「代入せず単純除外」方針で`is_invalid`に統合する
- alpha=1e-6の根拠: 全店舗のTier A行×チャンネル検定数(約10万件)に対するBonferroni
  補正後の有意水準(≈5×10⁻⁷)に近く、実データで確認したところ閾値をこの前後で動かしても
  除外される行の集合が変わらない(境界にp値の空白地帯が存在する)
- **理論値へのクリップ(Winsorize)は行わない**: 代入は虚構の値を作ることになり、
  判定不能の扱い方針(低設定・中間値を代入しない)と矛盾するため
- 発見の経緯: `assign_tier.py`が「BIG/REG列のみの確率表→無条件Tier A」としていたが、
  沖ドキ!シリーズ・ドッチ・ディスクアップULTRAREMIX等はBIG/REG列名の確率表を持つ
  AT/スマスロ機であり、これらはTier誤判定(後述)が根本原因だった。Tier判定を修正した後も
  キングハナハナ-30・マイジャグラーVの一部の行にサイト側の表記ミスとみられる異常が残り、
  この検定で対応する

**Stage 4-1: 欠損偏りガード**

判定不能が低稼働日に偏って発生するリスクに対応するガード機構。

```python
# 各検定の2群(候補日群/非候補日群)それぞれで算出
判定不能率 = 判定不能日数 / 全カレンダー日数

# 両群の判定不能率の差が閾値(仮: 10〜15pt)を超える場合
if abs(率_候補群 - 率_非候補群) > 閾値:
    → 「欠損偏りあり」フラグを立て、信頼度を下げる
    # 大幅超過の場合は検定自体をスキップ → 「検出不可」扱い
```

閾値の具体値は実データ運用後に調整(保留事項)。

### Stage 5: 複数店舗データでの精緻化(`multi_store.fit_hierarchical_model`、2026-07再設計)

階層モデルで店舗をランダム効果として扱い、重みを自動学習する。

$$\text{LogOdds}_i = \beta_0 + \beta_1 y_{1i} + \beta_2 y_{2i} + (\beta_3 + \gamma_{\text{store}(i)}) y_{3i}$$

- $\gamma_{\text{store}}$: 店舗固有補正項。繁盛度・客層・イベント文化の強さを代理表現
- $\gamma_{\text{store}}$ 自体がstage3スコアの精度向上と店舗プロファイル(セクション9)の特徴量として再利用される

**⚠ 旧実装の欠陥(2026-07判明・再設計)**: 旧実装は「y1を教師に学習したカーブ(y3)をy1に回帰」しており、デシル平均は二乗誤差最小の予測子なので**傾きが構成上ほぼ1になる循環学習**だった(実測β3≈0.985はこの理論値通りで、何も学習していなかった)。γ_storeもプールカーブのフィット残差=ノイズ主体だった。

**現実装(LOSO交差検証+直交化+合格ゲート、`validate_kaiten_channel`)**:
1. 店舗sを除いた残り店舗でbin_curvesと直交化パラメータを学習し、sにout-of-sample適用
2. y3をy1で直交化した残差 $y_3^{\perp}$ について、**同一台内で当日の$y_3^{\perp}$が翌観測日のy1を予測するか**を相関で検証(同日y1との相関は「出ている台に客が座り続ける」逆因果で汚染されるため、設定の据え置き傾向を介して翌日に伝わる成分を教師にする)
3. 合格条件: プール相関>0 かつ p<0.05 かつ 相関が正の店舗が70%以上 かつ 2店舗以上で評価可能
4. **合格時のみ** w3=プール回帰傾き(0〜1クリップ)、γ_store=店舗別傾き−プール傾き。**不合格時は w3=0.0(チャンネル無効)・γ_store=0.0**
5. 本番適用用のカーブ・直交化係数(orth_a/orth_b)は全店舗プールで学習して保存(検証はLOSO、適用はプールという分担。新規店舗にも適用するため)

**スケーラビリティ**:
- Stage1a(Tier判定・機械バイアス補正)は機種×データ提供元で1回校正 → 全店舗で再利用
- 新規店舗追加で個別学習が必要なのは $\gamma_{\text{store}}$ のみ

- 学習結果は`stage3_channel_weights.json`(`w1`/`w2`/`w3`/`orth_a`/`orth_b`)・`store_profile.gamma_store`(店舗ごと、分析DB`ホールデータ/analysis.db`)に保存される。`python multi_store.py`で全店舗一括実行(検証レポートも表示される)

### Stage 6: 検証戦略(教師ラベルなし。`multi_store.py`で実装済み)

1. **店舗間Tier再現性チェック**(`check_tier_reproducibility`): 同一機種が複数店舗にあれば、`judge_tier`の再判定結果とkaiten_zscore↔logLR_rngの店舗内相関の符号が店舗を跨いで一致するか確認(各店舗30行未満は対象外)
