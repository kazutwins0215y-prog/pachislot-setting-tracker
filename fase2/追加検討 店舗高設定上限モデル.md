# 追加検討 — 店舗内高設定台数の上限制約モデル

> このファイルは既存計画書への直接追記ではなく、独立した追記ファイルです。
> 各セクション末尾に「どのファイルのどこに統合すべきか」を明記しています。実装時にはこのファイルの該当箇所を参照ファイルへ転記・統合し、本ファイルは統合後にアーカイブしてください。

## 1. 背景・経緯

当初、以下5項目を店舗レベルの事前分布(prior)として設定推測に組み込む案を検討した。

1. ベース設定が1の店・2の店が存在する
2. 高設定割合はイベント日で2〜3割、5/6は0.5〜1割、通常営業日はそれ以下
3. 最高設定が5の店、ブラフで3を使う店が存在する
4. 店によって設定の量とバランスが異なる
5. 周年・グランドオープンを除き高設定が5割を超えることは稀

検討の結果、**項目1〜3(店タイプの主観分類)は「直感に頼りすぎるため確証データになり得ない」として却下**。一方で**項目5(高設定割合の上限)は「かなり確実性がある構造的制約」として採用**という結論に至った。

さらに議論を進め、以下の追加要件が判明した。

- 上限をこちらが固定値で設定すると、店舗ごとの配分差(例: イベント日で3割の店 vs 1割の店)により、緩い店では無意味・厳しい店では本物のシグナルを削るという非対称な悪影響が出る → **店舗ごとにデータから上限を推測すべき**
- 同一店舗でも設定配分は月によって変化しうる → **時間変化に追従する必要がある**
- 理想は「データが増えるほど精度が上がる」かつ「変化にも追従する」の両立

この2要件の両立という観点から複数のアルゴリズム候補を比較し、段階的な実装ロードマップに至った。

## 2. 追加する機能の位置づけ

### 2.1 何をする機能か

Stage3は台ごとに独立に `sigmoid(LogOdds)` を出力するため、店舗内で合計すると理論上ありえない水準まで高設定確率の期待値が積み上がることがある。これを検出し、店舗×日単位で確率を下方に補正(shrinkage)する後処理を追加する。

```python
# 店舗×日ごとに集計
E[高設定台数] = Σ high_prob(台)
N = 稼働台数
上限 = <店舗×日タイプごとに動的推定>  # 本ファイルの主題

if E[高設定台数] / N > 上限:
    offset = 二分探索等で E[高設定台数]/N が上限相当に収まるよう調整
    high_prob_calibrated = sigmoid(LogOdds - offset)
```

### 2.2 既存アーキテクチャ上の位置

- **処理順序**: Stage3(統合スコア化)の出力を、score.py が店舗プロファイルに書き込む**直前**の後処理ステップとして挿入する。patterns.py のS_全台系(検出用信号)とは役割が異なる点に注意(S_全台系=「多そうか」を検出する信号、本機能=「ここまでしかあり得ない」というキャリブレーションの上限)。
- **性質**: 個別台の事後確率の話ではなく、店舗×日単位の集計制約 = Stage5の階層モデル($\beta_0$切片)に対する**正則化**として位置づけられる。個々の店の癖(γ_store)は自由に学習させつつ、その上に構造的な天井をかぶせる。
- **ハードキャップ禁止**: 上限超過時は強制的に上限ちょうどに収める(ハードキャップ)のではなく、超過度合いに応じて連続的に縮小するソフトな正則化(ベイズ的shrinkage)とする。強さは`weights.json`と同様に設定ファイルで調整可能にする。
- **例外処理が必須**: 周年・グランドオープン等の既知の特異日を無条件にこの制約に通すと、本来検出したい「本物の強い日」を機械的に潰してしまう。Stage4-1(欠損偏りガード)と同型の罠であり、特異日の識別方法(外部日付指定 or 極端な乖離自体を「特異日候補」として別枠検出)を別途設計する必要がある(未解決・保留)。

## 3. 上限を「絶対上限0.5」の下でどう推測するか — 候補比較

前提: 5割は絶対に超えない上限として固定し、店舗ごとの実質的な上限(0.5以下のどこか)をデータから推測する。

| 候補 | 概要 | データ量が増えるほど精度UP | 月次変化への追従 | 実装コスト |
|---|---|:---:|:---:|:---:|
| A. スライディングウィンドウ分位点 | 直近Mヶ月のみで分位点算出 | × (Mで頭打ち) | ○ | 低 |
| B. 指数加重移動分位点(EWMA) | 古いデータを指数減衰 | △ | △(緩やかな変化のみ、不連続変化に弱い) | 低 |
| C. 既存の長期/短期αブレンド流用 | 長期版+短期版をウォークフォワードでα学習して合成 | ○ | △(店単位で1つのαに収束、単発の変化点は捉えない) | 低(既存コード再利用) |
| D. Kalman filter(状態空間モデル) | 真の上限を緩やかに動く潜在状態として逐次推定 | ○ | ○ | 高 |
| (参考)経験ベイズ縮小 | 店固有分位点と母集団分位点を履歴日数で重み付き平均 | ○ | ×(時間変化は非対応、店間差のみ吸収) | 低 |
| (参考)マクロ整合性逆算(Stage6拡張) | Tier A機種の理論/実測機械割の乖離から上限を逆算 | ○ | × | 中(Tier A機種必須) |
| (参考)極値理論(EVT/GPD) | 分布の裾をGPDでフィットし外挿 | ○(要大量サンプル) | × | 高(単店データではオーバースペック) |

### 3.1 各候補のメリット・デメリット詳細

**候補A: スライディングウィンドウ分位点**
- メリット: 実装が最も単純。月次変化への追従は直接的。
- デメリット: ウィンドウ幅固定で精度が頭打ちになる。既存の短期版サブスコアと同じ「サンプル不足時に不安定」という弱点をそのまま抱える。

**候補B: EWMA型分位点**
- メリット: 候補Aより滑らかに追従しつつ、減衰が緩やかなら長期データもある程度活用できる。
- デメリット: 月初の方針転換のような不連続変化への追従が1〜2週間程度遅れる。

**候補C: 長期/短期αブレンド流用**
- メリット: S_全台系・S_鉄板台と同一コード構造を再利用でき実装・検証コストが低い。αが店ごとに学習されるため「月ごとに大きく変わる店」は自動的に短期重視に収束する。
- デメリット: 単発の変化点そのものは検出しない(平均的なバランスを学習するのみ)。短期ウィンドウのサンプル不足時は既存方針通りα=0でフォールバックが必要。

**候補D: Kalman filter**
- メリット: 「データが増えるほど精度が上がる」と「変化への追従」を単一メカニズム(Kalmanゲインの自動調整)で両立できる。既に店舗パラメータの時変化用に同インフラを構築予定のため、実装基盤を共有できる。
- デメリット: プロセスノイズ/観測ノイズの分散比のチューニングが必要で、誤ると過剰追従または過少追従に転ぶ。単発の急変化に対して数日〜1週間のタイムラグが生じる。

## 4. 結論・推奨アルゴリズム

### 4.1 採用方式

**候補D(Kalman filter)を最終形として採用し、候補C(長期/短期αブレンド流用)を先行実装として段階導入する。**

```python
# 候補D の骨格
状態_t = 状態_{t-1} + プロセスノイズ   # 月次変化を緩やかなドリフトとしてモデル化
観測_t = 状態_t + 観測ノイズ           # その日の高設定割合
Kalman更新:
  不確実性が高い(データ少) → ゲイン大 → 新しい観測を強く反映
  不確実性が低い(データ多) → ゲイン小 → ノイズに惑わされにくい
上限 = min(0.5, 状態推定値 + 安全マージン)
```

### 4.2 採用理由

1. 要件(データ増加で精度向上 かつ 時間変化への追従)を1パラメータ(プロセスノイズ分散)で連続的に両立できる、他候補にはない特性を持つ。
2. 設計上すでにKalman filterを店舗パラメータの時変化に使う計画があるため、新規レイヤーではなく既存機構の適用範囲拡大にあたり、開発・検証コストを抑えられる。
3. データが少ない新規店では自動的にゲインが大きくなり「直近観測を素直に信じる」挙動になる。これはStage4-1・αブレンドの「データ不足時は段階的にフォールバック」という既存方針と自然に整合する。

### 4.3 段階的実装ロードマップ(推奨)

いきなり候補Dから作ると、未実装のPhase F(multi_store.py)側のKalman基盤に依存関係が生じ、単店運用中に検証できなくなる。そのため:

```
Step 1: 候補C(既存の長期/短期αブレンド流用)で最小実装
        → 単店運用中でも動作・検証可能
Step 2: multi_store.py 側で店舗パラメータ用Kalman filterが実際に構築された時点で、
        上限推定ロジックをそちらに統合(候補D化)
```

### 4.4 却下した項目とその理由(記録として残す)

- **店のベース設定(1/2)・最高設定(5/6)・ブラフの有無**: 観測データから直接検証できず主観に依存する。Stage5/Stage6の「教師ラベルなし・客観的整合性チェック」という設計思想と衝突するため、店舗プロファイルへの主観ラベルとしての組み込みは行わない。
- **固定値による上限設定**: 店舗間の配分差(イベント日2〜3割の店 vs それ以下の店)により、緩い店では無意味・厳しい店では本物のシグナルを削るという非対称な悪影響が出るため不採用。コールドスタート時のフォールバック値としてのみ限定利用する(4.5参照)。

### 4.5 データ不足時のフォールバック

```
その店・その日タイプで十分な履歴(信頼度ガードを通過) → Kalman/αブレンドによる動的上限
データ不足(新規店等)                                    → 業界一般値(0.4程度、要運用後調整)を暫定使用
```
