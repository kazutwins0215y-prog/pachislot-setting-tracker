## patterns.py

### 台同一性・移動/撤去/増台の判定ロジック

```python
# 店舗×機種名で D-1 と D を比較 (K=0日前日比較)
消失台番号 = {D-1に存在} - {Dに存在}
新規台番号 = {Dに存在} - {D-1に存在}
N = min(len(消失台番号), len(新規台番号))
移動 = N件  # S_移動台の対象
撤去 = 消失台番号[N:]
増台 = 新規台番号[N:]  # S_新台増台の対象
```

- 号機なし前提のため「どの台がどこへ移動したか」の1対1対応は不可。件数ベースでまとめて移動扱い
- 入替作業が複数日に分散した場合、撤去と増台が誤分離される可能性あり(現状許容、K拡張で緩和可)

### パターン検出: 幅(breadth)型

| パターン | データ | 検出時期 |
|---|---|---|
| 全台系 | Stage3スコアの当日横断集計(同機種/列/島) | 1日分でも検出可 |
| 新台/増台 | 増台後・直近移動平均 - 店舗内同機種全体平均 | 増台直後から自動フェード |
| 移動台 | 移動後・直近移動平均 - 店舗内同機種全体平均 | 移動判定ロジックのトリガー後 |

```python
# 新台増台・移動台の共通構造
短期トレンド = Stage3スコアの直近N件移動平均
基準値 = 設置/移動以降の累積平均 or 同機種店舗全体平均(Stage3スコア基準)
raw = 短期トレンド - 基準値
S_新台増台 or S_移動台 = np.clip(raw / 0.5, -1.0, 1.0)  # 符号付き(2026-07変更)
```

**符号付き化(機能B再設計Phase2.1、2026-07実装済み)**: 旧実装は`max(0, raw)`でクリップし
「強さ」のみを表現していたが、`max(0, ...)`を外して`np.clip(raw/0.5, -1, 1)`に変更。新台・移動台
がむしろ基準値以下に沈んでいる(＝弱い)ケースを負値で表現できるようにした。

### 機種強さ軸・全台系/高配分の日次判定(`score_zentaikei_judgment`、2026-07-09設計合意・Phase1実装済み)

`S_全台系`(平均×(1−std/0.5))は「揃い」を要求するため、機種内半分だけ高い日(1/2全台系相当)はstdが
大きくスコアが逆に下がり、全台系と鉄板台の谷間に落ちる問題があった(今後の実装予定.md 1.8節ギャップ2)。
機種×日×ホール粒度でzスコア・投入率・S_全台系の3指標を役割分担させて解決する。

```python
z = (Σhigh_prob - n×π) / √(n×π×(1-π))   # 異常検定(投入量が期待から何σ上振れか)
投入率 = Σhigh_prob ÷ n                  # 規模の推定(全台系≈1.0、1/2≈0.5)
S_全台系 = 既存score_zentaiki            # 揃いの確認(全台か部分かの形状判別)
```

- n=機種内有効台数(is_invalid除外)、π=`preprocess.load_channel_weights()['prior_high_ratio']`(Stage3のβ₀と同じ値を使う。ハードコードしない)
- Σhigh_probは較正済み事後確率の和＝期待高設定台数として意味を持つ(Stage3のβ₀導入の帰結)
- 判定ラベル(3値、`JUDGMENT_LABEL_*`定数): `z`が`JUDGMENT_Z_THRESHOLD`(=2.0)以上**かつ**同日×同ホール内の
  全機種で`benjamini_hochberg`によるFDR補正を通過した場合のみ「異常」とし、`S_全台系`が
  `JUDGMENT_ZENTAIKEI_S_THRESHOLD`(=0.5)以上なら**全台系**、未満(またはNaN=機種内有効台数<2で算出不能)
  なら**高配分**とする。異常でない場合は**普段どおり**(S_全台系は見ない)
- 投入率は生の比率(0〜1)で返す(表示側の×6換算は呼び出し元(機能A)の責務。3節の正解発表ラベルの
  投入率=k/nと直接比較できる粒度で保存するため)
- 閾値は固定値でPhase1開始→`machine_judgment_log`(下記データモデル参照)への蓄積後、実測(将来は3節の
  正解発表ラベル)と突き合わせてPhase2で調整する(wᵢ学習=1.6節と同じ「記録側先行」パターン)

### 少台数機種向けFisher版(`score_zentaikei_judgment_fisher`、2026-07-14実装・検証で不採用方向・並走記録のみ継続)

上記z検定がn=2〜4の少台数機種で検出力不足(high_probが0.6程度で飽和しz≥2に届かない)という
問題への対応として実装した代替判定。各台を`high_prob>=θ`(θ=`JUDGMENT_FISHER_THETA`=0.5)で
hot/notに二値化し、機種内hot数kを「同日・同ホールの店内他台(自機種除く)のhot率」を対照群とした
片側Fisher直接確率検定で評価する(対照群を同日店内実測にすることで店舗オフセットを自動吸収)。
判定ラベルはFDR有意かつk==nで「全台系」、k<nで「高配分」の3値スキーマを踏襲。
`run_store_profile._run_zentaikei_judgment_fisher`が`machine_judgment_fisher_log`へ並走記録する
(表示・合成には一切使わない。既存の`score_zentaikei_judgment`/`machine_judgment_log`は無変更)。

- **n=1(`JUDGMENT_FISHER_MIN_N`=2未満)は検定せず、FDR補正の家族からも除外する**。実装時に
  p=1.0のダミーとして家族に含めたところBH分母mだけが増えて実在候補の検出力を潰す過剰補正が
  発生(東中野で全期間有意1件)。テストしない仮説は最初から家族に入れないのが正しい設計
- **検証結果(2026-07-14、東中野の正解発表12件との突合)**: recall Fisher 0/13・現行z 3/13で
  **不採用方向が確定**。二値化が「どれだけ高いか」を捨てるため小標本のp値下限(n=2で約0.03)が
  日内FDR(1日25〜67機種)を数学的に突破できない。正解発表との突合で判明した構造
  (全6でも兄弟台は差枚マイナスが普通・13件中10件は1台だけの爆発・FDRが本物を2件殺す等)と
  次の設計(識別モードv1、実装保留)は`今後の実装予定.md`1.8.1節が一次情報

### 末尾版: 台番号末尾グループのカレンダー構造検出(`build_group_calendar_conditions`、2026-07-10設計・実装済み)

機種強さ軸と同じ3役分担の機械を、グループ定義を「機種」から「台番号末尾(店舗内全台横断)」に
差し替えて適用する(今後の実装予定.md 1.8節「末尾版」)。発端は「マルハン新宿東宝・エスパス歌舞伎町は
6のつく日に台番号末尾6が高配分」というユーザードメイン知識で、フェーズ1(検証スクリプト)で実データ
答え合わせを行ってから実装した。

**フェーズ1検証で判明した重要な事実**: 当初設計のStouffer統合z(`z=(Σp−nπ)/√(nπ(1−π))`を
`Σz÷√k`で統合)は、店舗全体の平均high_probがπより系統的に高い(sigmoid飽和の右裾。中央値は
π付近だが平均は台単位で0.24〜0.26まで押し上がる)ため、実データで11グループ×日付条件の
**全198仮説が有意化して閾値として破綻**した。既存`calendar_test`(鉄板台のカレンダー検定)と
同じ「候補日 vs 対照日」の相対検定に変更することで店舗オフセットが自動的に相殺されると判明し
(3案を実データ比較し、A案=Mann-Whitney Uが偽仮説(「末尾6×金曜」等の部分一致漏れ込み)を
効果量ゲートで唯一抑制できたため採用)、以下は変更後の確定仕様。

```python
グループ = tail_digit_group(台番号)   # 末尾0〜9 + ゾロ目(全桁同一: 11,22,…,99,111,…)の11区分
投入率(グループ×日) = Σhigh_prob ÷ n  # is_invalid除外

# レイヤー2本体: グループ×日付条件ごとに「候補日 vs 対照日」のMann-Whitney U片側検定(greater)
group_calendar_test(df, hole_name, group_series)
  → 全49候補(calendar_candidates)×11グループ = 539仮説の(該当日数,対照日数,p_raw,効果量)

# 一致ルール2本: 固定グループでは表現できない動的な対応(日によって「一致する台」が変わる)を
# 日ごとの「一致 vs 非一致」投入率差の対応ペアに対するWilcoxon符号順位検定(片側greater)で検定
match_rule_test(df, hole_name, rule='下2桁一致')  # 日付の日(1〜31)==台番号下2桁(unit%100)
match_rule_test(df, hole_name, rule='末尾一致')    # 日付の日の末尾(day%10)==台番号末尾(unit%10)

# 統合エントリポイント: 上記541仮説(539+2)を1つの仮説群としてBH補正+効果量ゲート(rbc>=0.3)
build_group_calendar_conditions(df, hole_name, group_series)
```

- グループ定義は関数注入方式(`group_series`を外部から渡す)。「グループ=機種」等への差し替え
  (今後の実装予定.md 1.8節「機種単位の癖分析」)でも検出ロジックは共通で再利用できる
- 最低該当日数: 候補日・対照日ともに5日未満の組み合わせは検定対象外(品質ガード兼・仮説数の
  自動削減。毎月X日条件は月1回のため5ヶ月分の蓄積が必要な点は許容済み)
- `calendar_candidates`は本節の実装で17候補→**49候補**(曜日7+末尾10+ゾロ目1+毎月X日31)へ
  拡張済み。既存`calendar_test`(鉄板台検定)も同じ関数を共用するため自動的に49候補で動作する
  (鉄板台の検出結果が従来と変わり得る点はユーザー許容済み)
- **重複条件の統合(max効果量)は保存時に行わない**: 「毎月6日⊂末尾6⊂一致ルール」のように条件が
  入れ子になるが、`group_calendar_conditions`テーブルには全条件をそのまま保存し、
  重複統合は予測時(下記`predict_tail_group_next_day`)にのみ適用する(検出は緩く保存・
  使用側でゲートの方針)
- **実データ検証結果(2026-07-10、マルハン新宿東宝・エスパス歌舞伎町)**: 本命「末尾6×6の日」が
  両店舗で541仮説中1位(rbc 0.98〜1.0)で検出。副産物として**ゾロ目台×ゾロ目の日(11日・22日)**
  (rbc 0.76〜0.78、ユーザー確認済み「末尾6と同等の鉄板」)、一致ルール2本も両店舗で有意
  (p<0.0001)を確認

**翌日予測(`predict_tail_group_next_day`、並走記録のみ)**: 台番号が属する末尾グループの
固定グループ×日付条件、および一致ルール(unit依存の動的判定)の両方について、翌日が該当する
有意条件を集め、**max(効果量)を採用**(重複統合。加算しない=二重計上回避)。`run_store_profile.py`
の`_run_tail_group_predictions`が予測種別`'S_末尾'`で`prediction_log`へappend-only記録する
(既存`write_prediction_log`の重複ガードを流用)。S_鉄板台等と異なり長期/短期のブレンドは行わない
(検出窓は全期間で開始する設計のため、使用alpha=0.0固定・長期スコア=ブレンド値として記録)。
**合成スコア(狙い目度)へは混ぜない**。`prediction_accuracy`で的中実績を確認してから合流を判断する
(遷移予測と同じ運用)。

### 機種版: 看板機種＋機種カレンダー癖の検出(`machine_group`/`group_constant_test`、2026-07-10設計・実装済み)

末尾版と同じ検出器(`group_calendar_test`/`build_group_calendar_conditions`)を、グループ定義
「機種」で再利用する(今後の実装予定.md 1.8節「機種単位の癖分析」)。対象は看板機種(恒常的に
強い機種)＋機種カレンダー癖(特定日にこの機種が強い)の2つ。新台/増台/再導入/移動台の
「導入後の扱い」は別軸の検討として今後の実装予定.mdに設計メモのみ残し、未実装。

```python
グループ = machine_group(df)  # 機種名そのまま(接頭辞なし)。日次有効台数の中央値が
                              # MACHINE_GROUP_MIN_UNITS(=2)未満の機種はNaN(検定対象外)

# 看板機種検定: 「自グループの投入率 - 同日の他グループ平均投入率」の対応ペアに対する
# Wilcoxon符号順位検定(片側greater)。日付条件='恒常'固定の1行を返す
group_constant_test(df, hole_name, group_series)

# 統合エントリポイント: group_calendar_test(機種カレンダー癖)+group_constant_test(看板機種)を
# 1つの仮説群としてBH補正(一致ルールは機種に意味がないため含めない)
build_group_calendar_conditions(
    df, hole_name, group_series, group_type='機種',
    include_match_rules=False, include_constant=True,
)
```

- **看板機種検定は末尾版の検定方式(Mann-Whitney U「候補日vs対照日」)をそのまま使わない**:
  「恒常的に強いか」は候補日/対照日の区分がなく毎日が該当するため、一致ルール
  (`match_rule_test`)と同じ「対応ペアのWilcoxon符号順位検定」を転用した。店舗全体の平均投入率が
  定数オフセットとして乗る問題(末尾版フェーズ1のStouffer破綻と同型)を、「自グループ vs
  同日の他グループ平均」の対応ペア化で回避する(2026-07ユーザー合意: 案A)
- **2窓検定並走(2026-07ユーザー合意: 案A)**: 恒常窓(全期間)と直近`RECENT_TEST_WINDOW_DAYS`
  (=90)日窓の2本を独立にBH補正して`group_calendar_conditions`へ保存する
  (グループ種別`'機種'`=恒常窓・`'機種_直近'`=直近窓)。個別台側の鉄板台等が採用する
  「検定は全期間1本+予測時に長期/短期αブレンド」とは異なる方式で、機種の癖は個別台より
  非定常(推し機種の月替わり)なリスクが高いと判断し、条件自体が消えたことを表現できる
  2窓検定を選んだ(実証後に個別台側へ展開するかは別途判断。今後の実装予定.md参照)。
  直近90日窓では毎月X日条件がほぼ検定不能(該当日数<5)になるが、これは想定内の挙動
  (既存の最低5日/5日ゲートがそのまま吸収する)
- **n≥2ゲート(2026-07ユーザー合意)**: 検定対象は日次有効台数の中央値2台以上の機種のみ
  (`MACHINE_GROUP_MIN_UNITS`)。2〜3台構成のローテは機種集計の恩恵が薄いため対象外とし、
  1.2節の台粒度ローテに任せる。「検出は緩く保存・使用側でゲート」の方針どおり、機種台数中央値は
  `group_calendar_conditions.台数中央値`列に保存し、表示/予測側の追加ゲート(暫定n≥3等)は
  この値を見て別途判断する(末尾版にはこの列は不要なためNULLのまま)
- **実データ検証結果(2026-07-10、マルハン新宿東宝・bigディッパー東中野)**: マルハン
  (検定対象97機種、全期間窓4850仮説中9件有意・直近90日窓4250仮説中3件有意。看板機種として
  北斗の拳/沖ドキ!ゴージャス/いざ番長等8機種+末尾8のカレンダー癖1件を検出)、bigディッパー
  (検定対象36機種、全期間窓1800仮説中1件有意=「いざ！番長」の看板機種)。効果量(rbc)は
  0.3〜0.6の範囲に収まり、末尾版フェーズ1で見られたStouffer破綻(全仮説有意化)は再現しなかった
- **DELETEスコープの修正(2026-07)**: `write_group_calendar_conditions`は従来「ホール名のみ」で
  当該ホールの全グループ種別を無条件に削除していたため、機種版を書き込むと末尾版の既存行が
  巻き込まれて消えるバグがあった。呼び出し側が`group_types`引数で削除対象を明示する方式に変更
  (省略時はresult_dfのグループ種別列から推定するが、result_dfが空になり得る呼び出しでは
  推定不能になるため明示が必須)

**翌日予測(`predict_machine_group_next_day`、並走記録のみ)**: `predict_tail_group_next_day`の
機種版(一致ルールに相当する動的グループがないためその分岐は持たない)。機種が該当する有意条件
('恒常'固定行は常に該当、カレンダー条件は翌日該当時のみ)を集め、max(効果量)を採用(重複統合)。
`run_store_profile.py`の`_run_machine_group_predictions`が恒常窓・直近窓それぞれ独立に
予測種別`'S_機種'`/`'S_機種_直近'`で`prediction_log`へappend-only記録する(機種→台展開、
台単位で書くのは`evaluate_predictions.py`が実測差枚と台単位で突合するため)。長期/短期の
ブレンドは行わない(使用alpha=0.0固定)。**合成スコア(狙い目度)へは混ぜない**。

### 機種バイアス除外・案A/案B(`identify_machine_bias`/`multi_store.learn_machine_bias_delta`、2026-07-14設計・実装済み)

2026-07-14の軸総点検で、機種検定(`group_constant_test`)の「恒常」判定が鉄拳6(11店中9店)・
いざ番長(11店中8店)で有意になっており、ユーザーの実地実感がないことから、店の癖ではなく
**機種側のhigh_prob推定バイアス**(スペック表由来の判定の甘さ等)が店舗横断で写っていると判断
(今後の実装予定.md 1.8.5節)。S_機種のspearmanが負寄りだった有力原因でもある。ユーザー要望
「機種が違っても同じ設定6は同じ強さとして検出してほしい」を受け、対症療法(案A)と根本対応の
並走実験(案B)の両方を実装した。

**案A(バイアス除外、合成済み)**: `identify_machine_bias(constant_conditions, total_store_count)`が
全店舗の`group_calendar_conditions`(グループ種別='機種'、日付条件='恒常')を集計し、機種ごとに
「BH有意だった店舗数 ÷ 全店舗数(プロファイル済み店舗数)」を計算、過半数超を「バイアス機種」と
判定する。分母は機種の在籍有無に関わらず全店舗数を使う(「8/11店」という書き方に合わせる設計)。
判定は恒常窓(全期間、グループ種別='機種')のみで行い、結果は機種/機種_直近の両方の恒常行に適用する
(バイアスは機種側の推定バイアスであり時間窓に依存しないため)。日付条件付き(末尾8・毎月X日等)は
店固有性が高いため除外対象にしない。

```python
bias_df = identify_machine_bias(constant_conditions, total_store_count)
# columns: 機種名, 対象店舗数, 有意店舗数, 有意店舗比率, バイアス判定
```

`run_store_profile.py`の`_refresh_machine_bias_list`が店舗処理のたびにこの集計を再実行し
`machine_bias_flags`テーブル(グローバル1本・全削除→再挿入)へ保存、バイアス機種名リストを返す
(集計は軽量なSQL集約のみのため店舗ごとの再計算でも実害はない。複数店舗一括実行では
ループが進むほど直近の検定結果を反映する)。`_run_machine_group_predictions`が、通常の
S_機種/S_機種_直近に加えて、有意条件からバイアス機種の「恒常」行だけを除外した変種を
S_機種_除外/S_機種_直近_除外として**同一期間に並走記録**する(除外前後を`prediction_accuracy`で
対比較するため、既存のS_機種は変更せず残す)。`app_b.py`の`_load_condition_significance`/
`_load_future_calendar_conditions`(おすすめ店舗スコア・有効性マトリクス・カレンダー投影が
再利用)も、バイアス機種の「恒常」行を集計・投影から除外する。

**案B(根本対応・δ較正、並走評価専用)**: `multi_store.learn_machine_bias_delta`が全店舗プールで
「自機種の日次投入率 − 同店・同日の他機種平均投入率」の対応ペア差分をlogit空間で計算し、
機種ごとに平均(raw_delta)を求めたうえで縮小推定(`delta = raw_delta × n/(n+SHRINKAGE_K)`,
`SHRINKAGE_K=200`)する。nが少ない機種を0へ引き寄せる、他のStage5学習と同じ思想。
`fit_hierarchical_model`実行後(bin_curves/channel_weights学習済み)に呼び、学習結果を
`machine_bias_delta.json`へ保存する(1回学習・全店舗再利用、bin_curves/channel_weightsと同じ
パターン)。`run_store_profile.py`の`_run_machine_bias_calibrated_predictions`がこのδで
log_odds(`log_odds − delta`)を較正したhigh_probを使い、機種恒常検定→翌日予測を
S_機種_較正としてprediction_logへ並走記録する(グループ種別='機種_較正'。**本流のhigh_prob・
合成スコア・表示には一切使わない**)。δ未学習時は記録自体をスキップする。

**評価**: 新規テーブル・予測種別を追加しただけで`evaluate_predictions.py`(予測種別ごとに汎用集計)・
`write_prediction_log`の重複ガード(ホール名/予測種別/使用データ最終日キー)はコード変更不要で
対応済み。S_機種・S_機種_除外・S_機種_較正の3系列を`prediction_accuracy`で比較し、
0節の検証ゲートに従って除外/較正の採否を判断する(今後の実装予定.md参照)。

### 導入後カーブ: 新台/増台/減台/再導入/純移動の判別と経過日数ビン検定(`detect_introduction_events`/`introduction_curve_test`、2026-07-13設計・実装済み)

新台・増台・移動台に共通して重要なのは「導入初日に入るか」ではなく**「導入後の一定期間、
どう扱われるか」**(店によって新台は弱いが増台は強い、等の非対称性がある)という
ユーザードメイン知識に基づく(今後の実装予定.md 1.8.3節)。旧`detect_events`(K=0前日比較のみ、
移動台検出専用)は変更せず、機種レベルのイベント判別・カーブ検定を行う新関数を追加した。

```python
# イベント判別: 「店舗観測日」(ホール内全機種横断のユニーク日付)を基準タイムラインとし、
# 機種ごとに台番号集合が非空だった日だけを辿って前回在籍日との差分を見る(欠測日は自然にスキップ)
detect_introduction_events(df, hole_name)
  → 日付, ホール名, 機種名, カテゴリ, 台数変化, 移動フラグ, 台番号リスト, 移動台番号リスト, 不在日数

# カーブ検定: カテゴリ5種×経過日数ビン5種(初日/2〜3日/4〜7日/8〜14日/15日以降)=最大25仮説を
# 店舗単位でBH補正(Wilcoxon符号順位検定・両側・符号付きrank-biserial効果量)
introduction_curve_test(df, hole_name, events_df)
  → グループ種別='導入後', グループ=カテゴリ名, 日付条件=経過ビン名, ...BH有意
```

**イベント判別のカテゴリ**:
- **新台**: 機種の店舗初出日(初出日＞店舗収集開始日の場合のみ。初出日＝店舗収集開始日は
  左打ち切りのため`判別不能`とし、カーブ学習・タブ表示のどちらからも除外)
- **増台**: 前日在籍あり・純台数増(復帰ガード適用後)
- **減台**: 純台数減。**7日(店舗観測日)遅延確定**(消えた台が以後7日間戻らないことを
  全数確認できた場合のみ確定記録。日次実行では毎回全履歴を再計算するため、直近の減台候補は
  未来日が足りず自然に保留され翌日以降の再実行で確定する。全撤去(機種消滅、以後二度と
  戻らない)は対応する「後」の系列が無いため対象外だが、再導入判定の不在日数起点としては
  引き続き機能する)
- **再導入**: 不在≥7日(店舗観測日ベース。店舗ぐるみの収集欠測を不在と誤認しない)を挟んだ再出現
- **純移動**: 台数±0で台番号変更あり(移動した台番号は`移動台番号リスト`に保存)
- **欠測ノイズガード(増台側)**: 直近7日(店舗観測日)以内に在籍していた台の再出現は「復帰」として
  増台の対象から除外する(旧`detect_events`はこれを増台/移動と誤カウントしており、旧`S_新台増台`
  にもノイズとして混入していた)
- **主軸=純台数変化、移動は付随フラグ**: 増減を伴う台番号変更(例: 4台→6台で全台番号変更)は
  独立カテゴリにせず増台/減台/純移動の**移動フラグ**として保存する(細かく保存→後で束ねる/
  層別するのは自由の原則)

**カーブ検定の設計**:
- 対象カテゴリごとに「同一機種の次のイベント(カテゴリ問わず)の前日」または(次イベントが
  無ければ)データ末尾までを追跡ウィンドウとする(次イベントの影響を古いイベントのカーブに
  混ぜないため)
- 新台/増台/減台/再導入(機種単位): その日の機種投入率(Σhigh_prob/n) − 同日の他機種平均投入率
  (対応ペア、`group_constant_test`と同じ「自グループvs同日他グループ平均」方式)。純移動のみ
  台単位: 移動台のhigh_prob − 同日店舗全体平均(台単位、投入率だと薄まるため)
- 店舗内の同カテゴリ全イベントの差分を1つのサンプル集団にプール(「店舗単位で学習」。
  「新台が強い店は毎回強い」というドメイン知識の前提と整合)
- **両側検定**(既存の看板機種/末尾版の片側greaterと異なる): 「この店は新台に入れない」という
  負のカーブも回避情報として検出するため。BH有意ゲートは|効果量|>=`EFFECT_SIZE_THRESHOLD`で
  正負どちらも拾う
- 対照は素の対店内差(A案)で開始。15日以降ビンも検定・保存する(看板機種交絡の兆候検知用。
  15日以降で有意=看板機種の恒常的な強さと導入後効果が混ざっている可能性を示唆)

**翌日予測(`predict_introduction_next_day`、並走記録のみ)**: 末尾版/機種版と異なりカテゴリは
店舗単位で1本しかないため重複統合ロジックは不要。`run_store_profile.py`の
`_run_introduction_predictions`が機種ごとの最新イベント(判別不能を除く5カテゴリのうち日付が
最も新しいもの。カーブ検定のウィンドウ打ち切りと同じ考え方)を特定し、翌観測日までの経過日数が
14日未満(15日以降ビンは予測対象外)ならBH有意な条件から効果量を取り出し、予測種別`'S_導入後'`で
`prediction_log`へ追記する。純移動はイベント時に実際に移動した台のみ、それ以外4カテゴリは
使用データ最終日時点でその機種に在籍する全台へ同じ予測値を書く(カーブ検定の対象粒度に合わせる)。
**合成スコア(狙い目度)へは混ぜない**(S_末尾/S_機種と同じ運用)。

**サブスコア分割はしない(最小案)**: `S_新台増台`/`S_移動台`(実績追従型のトレンド−基準値)は
現状維持。カテゴリ情報はカーブ学習＋`S_導入後`並走＋トップページ新台/増台タブの分離
フィルタ(下記)のみに使う。

**トップページ新台/増台タブの分離(`app_top.py`、実装ステップ4)**: 旧`detect_events`は
「機種の店舗初出」と「既存機種の台数追加」を区別できず、新台/増台タブは同一の`S_新台増台`
データを表示していた(今後の実装予定.md 1.2a節)。`introduction_events`テーブル(下記データ
モデル参照)から(ホール名, 機種名)ごとの最新イベントカテゴリを読み、新台タブ=最新イベントが
`新台`の機種、増台タブ=`増台`の機種のみを表示するよう振り分けるフィルタを追加した
(スコアリング自体は`S_新台増台`のまま変更なし)。旧`detect_events`由来の`S_新台増台`に
混入していた復帰ノイズ(欠測ガード未対応の台)は、`detect_introduction_events`のカテゴリに
該当しない機種として両タブから自然にふるい落とされる。

**二重計上の宿題(未解消・既知のリスク)**: 増台に伴い移動フラグが付いた台は`S_移動台`と
`S_導入後`が重複し得る。並走記録の段階(予測種別が別)では許容し、将来の合成合流時に
排他/階層設計が必要(「設計上の合意事項」の交絡リスクと同じ扱い)。

**既知の限界**: 機種名の表記ゆれ・後継機での系列切れに再導入判定が特に敏感(名寄せは対応外)。

### パターン検出: 深さ(depth)型

深さ型は**履歴日数が増えるほど精度が上がる**。データ不足時は「検出不可」扱い(低スコアは出さない)。

#### 鉄板台の検出フロー(2経路)

**メイン経路: 未知周期の発見**

```
Step1: Stage3スコア系列を欠損込みの日次グリッドで保持
Step2: ACF(pairwise-complete版、欠損ペアを除外)で lag 1〜60日をスクリーニング
Step3: PDM(Phase Dispersion Minimization)で波形非依存の再確認
       (ACFは線形相関が前提のため矩形的な鉄板台パターンへの感度が弱い)
Step4: 判定不能率が高い台(欠損が多い場合) → Lomb-Scargle periodogramに切り替え
       (等間隔サンプリングの前提が崩れているため)
```

**別経路: 既知カレンダー条件の検定**

```
候補49カテゴリ(`patterns.calendar_candidates`。2026-07-10「末尾版」フェーズ2で17→49候補へ拡張):
  曜日:    月/火/水/木/金/土/日 (7候補)
  日付末尾: 0〜9 (10候補)
  ゾロ目:  該当/非該当 (1候補)
  毎月X日: 1〜31日 (31候補、末尾版レイヤー2との共用のため追加)

各候補で:
  1. 一方向(one-sided)の並べ替え検定 or Mann-Whitney U検定
  2. p値を Benjamini-Hochberg FDR補正 (候補49 × 対象台数)
  3. 補正後有意 かつ 効果量(中央値差/rank-biserial相関)が閾値以上 → 候補として残す
  4. Stage4-1の欠損偏りガードを必ず通過させる
```

> 候補拡張(17→49)に伴い鉄板台のカレンダー検出結果が従来と変わり得る点、仮説数増による
> BH検出力の微減はユーザー許容済み(今後の実装予定.md 1.8節「末尾版」参照)。

> **rank-biserial相関の符号に注意(2026-07バグ修正)**: `calendar_test`(patterns.py)は
> 一方向検定`alternative='greater'`(候補日が対照日より高いかを検定)のp値と、
> 効果量`rank-biserial`相関を組み合わせて有意判定するが、`rbc = 1.0 - 2.0*u_stat/(n1*n2)`
> という符号のまま実装すると、p値が検定通り有意でも「候補日が高い」ケースほど`rbc`が
> **負**になり、正の閾値`effect_size>=0.3`のゲートで常に弾かれてしまう。修正版は
> `rbc = 2.0*u_stat/(n1*n2) - 1.0`(候補日が対照日より高いほど正になる)。人工的な
> 強い曜日パターンで検証したところ、修正前はp=1.2e-17でも有意判定されなかったが
> 修正後は正しく有意判定された。修正前は全店舗でカレンダー有意が1件も検出されず、
> S_鉄板台が理論上の最大値1.0(メイン経路・カレンダー経路の両方一致)に到達できていなかった。

**スコアの付与方法(2026-07仕様変更: 該当日のみ・効果量ベース→同年再設計で符号付き化)**:

旧実装は検出台の**全日**に定数(両経路一致=1.0/片方=0.6)を付与していたが、「特定条件の日に入る」という鉄板台の性質と逆に、非該当日の狙い目度を押し上げて該当日のコントラストを消していたため、まず該当日のみ効果量ベースでスコア付与する方式に変更し、続く機能B再設計(Phase2.2、2026-07実装済み)で非該当日を「弱さ」として符号付き表現するよう再拡張した。

```
カレンダー経路: 有意候補(例: 末尾_7)に合致する日 → 効果量(rank-biserial、0〜1クリップ)を正のスコアに
               合致しない日(この機種にカレンダー経路の検出が1つ以上ある場合のみ)
               → -NEGATIVE_SCALE × 検出済み効果量の平均(負、NEGATIVE_SCALE=0.5暫定値)
周期経路:      確認済み周期の位相ビン(5ビン、pdm_confirmと同一)ごとの平均high_probを求め、
               全体平均を上回るビン(該当ビン) → (ビン平均−全体平均)/0.5 を正のスコアに
               それ以外のビン(非該当ビン) → -NEGATIVE_SCALE × 該当ビン効果量の平均(負)
               (位相は観測順インデックス基準。営業日が飛ぶと暦日とズレる既知の近似)
2経路統合:     符号付きnoisy-or(`_combine_signed`)。正の値同士は通常のnoisy-or、負の値同士は
               `-(1-(1-|a|)(1-|b|))`、符号が異なる場合は単純平均(両経路が同じ方向を指すほど
               絶対値が大きくなる。旧「両方一致→1.0」の連続版を符号付きに拡張したもの)
検出自体がない台: 全日NaN(synthesizeで除外・再正規化。判定不能を負値にすると新規台が
               不当に低評価になるため、Stage4-1と同じ方針で「検出不可」のまま維持する)
```

検出条件(経路/条件/効果量/周期経路は周期日数)は`score_teppandai(details_out=...)`で台単位に取り出せ、`run_store_profile.py`が分析DBの`teppan_conditions`テーブルに保存する(「明日は該当日か」の判断材料。トップページ表示・翌日予測(下記)で使用)。

#### 遷移モデル(据え置き/上げ/下げ)による全台翌日予測(Stage7-3、2026-07-07実装)

設定運用は「据え置き」だけでなく「上げ」「下げ」を含むため、翌日予測の事前分布を店舗単位の2状態(高/低)マルコフ遷移として持つ。S_鉄板台の翌日予測(検出済みの台のみ)と異なり、**使用データ最終日に判定可能だった全台**が対象(全台カバレッジ)。

```
p_stay = P(高_t | 高_{t-1})   … 据え置き率(1 - p_stay が下げ率)
p_up   = P(高_t | 低_{t-1})   … 上げ率
翌日予測 = 当日high_prob × p_stay + (1 - 当日high_prob) × p_up
```

- **推定(v1: 無条件版)**: 真の設定ラベルは観測不能のため、連続した暦日ペア(同一ホール×機種×台番号、両日とも判定可能)の事後確率high_probのソフトカウントで推定(`patterns.estimate_transition_matrix`。暦日差1日でないペアは除外、最低`TRANSITION_MIN_PAIRS`=50ペア)。事後確率のノイズにより持続性を過小評価する既知のバイアスあり
- 長期(全履歴)・短期(直近`SHORT_WINDOW_DEFAULT`日)の両方で遷移行列を推定し、S_鉄板台予測と同じ規約(`FIXED_ALPHA`ブレンド・短期不可=α実質0)で`patterns.predict_transition_with_blend`がブレンドする
- `run_store_profile.py`の`_run_transition_predictions`が上限キャリブレーション補正後のhigh_probを入力に、予測種別`'遷移予測'`で`prediction_log`へ追記(append-only・リーク禁止はS_鉄板台予測と同一規約)。`evaluate_predictions.py`は(ホール名, 予測種別)で汎用集計するため無改修で精度計測に乗る
- `app_top.py`の翌日予測ランキングは`予測種別='S_鉄板台'`のみ表示(レンジ・意味の異なる2種の混在回避。遷移予測の表示は精度確認後に検討、[`今後の実装予定.md`](今後の実装予定.md)1.4節5項)

**実装時の実データ検証で判明した重要な事実(2026-07-07)**: 全6店舗で遷移行列がほぼ縮退(p_stay≈p_up≈店舗平均high_prob。例: yasuda7でp_stay=0.275/p_up=0.273)しており、**これらの店舗は日次でほぼ無記憶**である。

- 前日high_prob↔当日high_probの自己相関(連続暦日ペア、n=1.5万〜12.7万/店): pearson 0.007〜0.055
- 当日high_prob→**翌日**実測差枚のSpearman(全履歴): -0.021〜+0.005(ほぼゼロ。エスパス日拓新宿歌舞伎町店は-0.021, p=2.6e-14で有意な**負**=「出た台は翌日出ない」傾向)
- 同日Spearman 0.77〜0.84との対比: 同日相関はチャンネル②が当日差枚から計算されることによる自己相関が大半で、翌日への持ち越し情報はほぼ含まれない
- 含意: (a)無条件の持続性ベース予測はこれらの店舗ではほぼ機能しない(v1の予測値は台間でほぼ定数になる)、(b)翌日予測の信号は「特定日・周期」(S_鉄板台の方向)と条件付き遷移(前日差枚→下げ、カレンダー→上げ、ローテ連動)に求める必要がある(拡張候補は[`今後の実装予定.md`](今後の実装予定.md)1.5節)、(c)この縮退自体が誠実な出力であり、評価ハーネス上で条件付き拡張との比較基準(ヌルモデル)として機能する
- **2026-07-13追記**: `prediction_accuracy`実測でも確認(`'遷移予測'`全店舗n=22,630でSpearman≈0〜負)。上記(a)の実測確定であり想定どおりヌルモデルとして機能している。全予測種別の実測精度サマリと対応方針は[`今後の実装予定.md`](今後の実装予定.md)4節「機能B理想形」の背景を参照

**店舗の癖(据え/上げ/下げ)の`store_profile`保存(2026-07-08 タスク3追記(c)。記録のみ、表示UIは今後の実装予定)**:

- `estimate_transition_matrix`の戻り値に`pi`(ベース率π＝ペア集合のp_prev平均。ソフトカウントと同じ集合で定義)を追加した
- `run_store_profile.run_for_hole`が長期版行列(`matrix_long`)を`update_store_profile`へ渡し、`store_profile`に`遷移_ベース率`/`遷移_p_stay`/`遷移_p_up`/`遷移_ペア数`の4列を追加保存する(gamma_store等と同じ「店舗スカラーを全パターン行に複製」方式)。ペア数不足(`matrix_long`がNone)の店舗は4列ともNULL
- 下げ率(=1−p_stay)・引き継ぎ度λ(=p_stay−p_up)は導出値のため保存しない(表示側で計算する想定)。設計メモ: 店舗横断比較はλを使う(p_stay生値はベース率πに引きずられ店舗間比較不可。λはπの影響が打ち消される共通スケール[-1,1]で、+1寄り=据え置き型・0付近=毎日リセット型・負=逆張り型)。店舗個別表示は据え癖p_stay/上げ癖p_up/下げ率1−p_stayの3値に「πとの差」を添える設計。ソフトカウントの平滑化バイアスにより推定ノイズが大きい店ほどλが0側に縮むため、僅差の順位比較は過信しないこと

#### 遷移モデルの前日差枚条件付き拡張＋並走ログ基盤(Stage7-3拡張、2026-07-08 タスク4実装)

無条件版(v1)は上記の実データ検証で「全店舗が日次でほぼ無記憶」と判明し予測力がほぼ期待できないため、「出た台は翌日出ない」逆信号(エスパス日拓新宿歌舞伎町店で当日high_prob→翌日実測差枚のSpearman -0.021、p=2.6e-14)を層別遷移行列で拾う条件付き拡張を追加した。

```python
def estimate_transition_matrix_stratified(df, hole_name) -> dict | None:
    """前日(t-1)の実測差枚が店舗内上位2割か否かで層別した遷移行列(v2: 条件付き版)"""
```

- **ペア構築の共通化**: 無条件版・層別版とも暦日差1日ペアの構築ロジックが同一のため`_build_transition_pairs`に抽出し二重実装を避けた(p_prev/p_curr/前日の日付/前日の実測差枚を持つDataFrameを返す)。p_stay/p_up/piの推定式(ソフトカウント)も`_fit_transition_from_pairs`に共通化
- **層の定義**: ペア(t-1→t)の前日(t-1)の実測差枚が、その店舗×その日(t-1)の上位`STRAT_QUANTILE`(暫定0.8)分位点以上か否かの2層(`stratify_threshold_by_date`で店舗×日の分位点を算出。閾値が店舗内の日次分位点のため店舗規模に自動適応。ロジスティック変調は不採用と決定済み)。差枚列(`slot_data`由来、preprocessでは加工されず素通りするため`scored`にも残る)が無い/全欠損の店舗は層別不可(None)
- **層ごとの最低ペア数**: `TRANSITION_MIN_PAIRS`(=50、無条件版と共通)未満の層があれば層別不可(None)
- **店舗別有効化(並べ替え検定)**: ソフトカウント由来のp_stay/p_upは通常の比率検定(z検定等)の前提(独立二項分布)に従わないため、層ラベルをシャッフルした帰無分布と観測差を比較する並べ替え検定(`_stratified_permutation_test`、`STRAT_PERMUTATION_ITERS`=1000回、有意水準`STRAT_SIGNIFICANCE_ALPHA`=0.05)を採用。統計量は`max(|Δp_stay|, |Δp_up|)`(p_stay・p_upのどちらか一方でも層間に有意差があれば層別に意味があるとみなす保守的な基準)。有意でない店舗は無条件版のみ(w3のLOSOゲート・1.6節wᵢ店舗別ゲートと同じ「信号なし店舗にノイズを持ち込まない」思想)
- **並走記録**: `run_store_profile._run_transition_predictions`が無条件版(予測種別`'遷移予測'`、全店舗・従来どおり)に加え、有意な店舗のみ条件付き版(予測種別`'遷移予測_前日差枚'`)を同じ実行・同じ台集合で追加記録する(DELETE/UPDATEなしの並走ログ)。条件付き版の予測は、その台の当日(使用データ最終日)の実測差枚が店舗内上位2割かで長期/短期それぞれの上位層・下位層行列を選び`predict_transition_with_blend`でブレンドする(条件に使う差枚は使用データ最終日の確定値のみのためリークではない)。層別短期行列が推定不可(ペア数不足)なら`predict_transition_with_blend`のブレンド規約どおり長期のみ(α実質0)
- **採否ゲート**: `evaluate_predictions.py`は(ホール名, 予測種別)ごと集計のため無改修で両種別を対比較できる(同一日・同一台集合でSpearman/Precision@Nを比較し、上回った場合のみ将来の表示採用を検討する運用)。`app_top.py`の翌日予測ランキングは`予測種別='S_鉄板台'`のみ表示のため新種別の混入なし
- **実データ検証(2026-07-08、9店舗)**: 層別が有意だったのはyasuda7(p=0.009)・グランパ中野(p=0.021)・マルハン新宿東宝ビル店(p=0.018)の3店舗。動機となったエスパス日拓新宿歌舞伎町店自体はこの層別検定ではp=0.134で非有意(層間のp_stay差は0.002程度と小さい)。**この並走ログの仕組みは以後の全拡張(カレンダー条件・ローテ連動・1.6/1.7節)の評価基盤として再利用する**ため、個々の店舗でゲート落ちしても投資は無駄にならない

#### 据え置きの日次判定と翌日予測(2026-07-08 タスク3実装)

旧`score_sueki`は台単位で全期間1定数(0〜1)を返しており、「日単位の該当/非該当」を表現できなかった。`patterns.score_sueki_daily`に差し替え、S_鉄板台と同じ符号規約([-1,1]・負=非該当日)の日次スコアにした(**案A「滑走窓自己相関」**。案B(CUSUM等の変化点検出)は段階投入の方針どおり今回は見送り)。

```python
def sueki_daily_r(hp: pd.Series) -> np.ndarray:
    """台ごとの直近K=14日窓のlag-1自己相関r_tを日次計算し、EWMA(span=7)で平滑化したr̄_tを返す"""
```

- 台ごとに直近`SUEKI_WINDOW`=14日窓でlag-1自己相関r_t を日次計算する。窓内の有効ペア(両日とも判定可能)が`SUEKI_MIN_PAIRS`=8未満の日はNaN(履歴不足)。10ではなく8にしたのは、14日窓(最大13ペア)にそのまま「10ペア」を適用すると窓の約8割充足が必要でNaN率が過大になるため(実測: 全8店舗・台×日約50万行で足切り8→NaN率20.5%/足切り10→30.7%)。1窓の相関のブレは後段のEWMA平滑がならす前提
- 窓内サンプルが少なくr_t自体がばらつくため`SUEKI_EWMA_SPAN`=7でEWMA平滑化する(`pandas.Series.ewm(span=7, min_periods=1).mean()`。先頭の連続NaN区間はNaNのまま、一度有効値が出た後の欠損日は直前の平滑値を保持する挙動)
- ベクトル化実装: 台ごとのpython/scipyループでpearsonrを都度呼ぶと約50万行規模で数十分オーダーの実行時間になる(実測: 50万回のscipy.stats.pearsonr呼び出しで約1650秒)。`pandas.Series.rolling(window=13, min_periods=8).corr()`はpairwise-complete(NaNペアはmin_periods判定から除外)なローリング相関をベクトル化して計算できるため、これを採用している
- **符号変換(`score_sueki_daily`)**: 平滑化後のr̄_tが`SUEKI_DAILY_THRESHOLD`(暫定0.2)以上の日を該当日(正スコア=+r̄_t)、それ未満の日を「切断」として非該当日とする。負スコアは`-NEGATIVE_SCALE × min(1, (SUEKI_DAILY_THRESHOLD - r̄_t) / SUEKI_DAILY_THRESHOLD)`(**2026-07-14応急処置**: 旧式はr̄_t=0で-1に飽和したが、S_鉄板台と同じ`NEGATIVE_SCALE`=0.5を乗じて緩和。無記憶状態(r̄≈0)の飽和値は-1→-0.5)
- **経緯(2026-07-08確認→2026-07-14応急処置実施)**: 旧負側式はr̄_t=0で-1に飽和し、全店舗が日次ほぼ無記憶(自己相関≈0、上記遷移モデルの検証参照)という実態のもとで大半の台×日が-1近傍・店舗平均`s_sueki`が大きく負(例: yasuda7で約-0.73)になっていた。`prediction_accuracy`実測(2026-07-13)で`'S_据え置き'`は全店舗Spearman≈0〜負(n=15,393)と確認され無効確定。これを受け2026-07-14に応急処置として①上記`NEGATIVE_SCALE`緩和、②`weights.json`のS_据え置き重み1.0→0.3への減重を実施した。**据え置き検出の実態分析(2026-07-14)**: 正スコア該当日は全店で台×日の10〜15%出るが、14日窓相関のノイズだけでも同程度が閾値0.2を超えるため大半が偽検出とみられる(予測が当たらないことと整合)。一方、遷移行列ではp_stayがベース率πを全10店で一貫して+1〜1.5pt上回っており(例: エスパス0.246 vs 0.233、リフト約1.05倍)、「据え置き癖は実在するが極めて弱い」が現状の結論。店舗の据え置き癖の監視役はp_stay−πに委ね、差が大きい店(据え置きイベント店等)が現れた時点で台単位検出のイベントベース再設計を検討する
- `compute_short_term_score`のS_据え置き分岐も同じ`score_sueki_daily`に差し替え済み(短期窓版)

**翌日予測(`run_store_profile._run_sueki_predictions`、`_run_teppan_predictions`の直後に実行)**:

```
翌日投影 = r̄_t ×(当日high_probの台基準からの偏差)
台基準 = その台のSUEKI_WINDOW(14)日窓内の平均high_prob
```

- 対象: 使用データ最終日にr̄_t(長期版)が計算できた(NaNでない)全台(検出条件に依存しないカバレッジという点はS_鉄板台と異なり遷移モデルに近い)
- 長期版(全履歴で計算したr̄_t)・短期版(直近`SHORT_WINDOW_DEFAULT`日窓で計算したr̄_t)を`patterns.predict_sueki_with_blend`が`FIXED_ALPHA`でブレンドする(短期不可=α実質0。S_鉄板台・遷移モデルと同じブレンド規約)
- `prediction_log`に予測種別`'S_据え置き'`で追記(append-only・リーク禁止は他の予測種別と同一規約)。`詳細`列に{r値, 切断フラグ, 当日偏差, 窓幅}を保存し、将来のwᵢ学習/GBDT融合(`今後の実装予定.md`1.6/1.7節)の特徴量として再利用できるようにしている
- `evaluate_predictions.py`は(ホール名, 予測種別)ごとの汎用集計のため無改修で`'S_据え置き'`も精度計測に乗る。`app_top.py`の翌日予測ランキング(個別台/機種タブ)は引き続き`予測種別='S_鉄板台'`のみ表示のため、新種別が混入することはない

**トップページ「据え」タブへの影響**: `S_据え置き`が「全期間1定数」→「当日断面の符号付き値」に変わったことに伴い、`app_top._SUEKI_SCORE_GATE`を0.5→0.0に変更した(該当日は必ず`SUEKI_DAILY_THRESHOLD`以上になる仕様のため、0超のフィルタで実質「最新日が据え置き該当日の台」を抽出する)。`app_b.py`は元々`range_y=[-1,1]`に対応済みのため無改修。

**店舗の「癖」指標の移管**: `store_profile`の`s_sueki`行スコア(=S_据え置き列の店舗平均)は、意味が「癖の強さ(0〜1定数の平均)」から「直近該当日の多寡(符号付き日次値の平均)」に変わった。店舗の据え置き癖を見る指標としての役割は、下記「遷移モデル」の`estimate_transition_matrix`が返す`pi`(ベース率)・`p_stay`・`p_up`(2状態マルコフ連鎖ではlag-1自己相関 r ≈ p_stay − p_up の関係にあり、癖とベース率を分離して保持できる上位互換)に譲る。

### 長期/短期αブレンド(Section6.3)

S_全台系・S_鉄板台・S_ローテ・S_据え置きの4種について、同一の検出ロジックを2系統で並行算出してブレンドする。

```python
# 同一の検出ロジックで入力範囲だけ変える
長期版サブスコア = 全履歴データを使った算出
短期版サブスコア = 直近Mウィンドウ(直近M日 or M件)のみを使った算出

# αは店舗・サブスコア種別ごとに個別学習
ブレンド済み = α × 短期版 + (1-α) × 長期版
```

**αの決め方(2026-07仕様変更: 固定値)**: 現在は固定α=0.3(`patterns.FIXED_ALPHA`)。

旧実装のウォークフォワード学習(`walk_forward_alpha`)は以下のリークがあり停止した:
- ターゲット(当日の店舗平均high_prob)に対し、特徴量のS_全台系等は**その当日のhigh_probから計算**されており、同日情報のリークで実質自己回帰だった(αが「未来の予測に効く比率」を表さない)
- `compute_short_term_score`は末尾M日窓を1回計算して貼るだけで各時点tの短期版になっておらず、α推定の実効サンプルが末尾のみだった

真のウォークフォワード(特徴=t時点までで計算、ターゲット=**t+1の実測差枚**)はStage7(予測精度の自己検証ループ)として機能B再設計とあわせて再実装する。`walk_forward_alpha`関数はその際の再利用に備えて残置(現在未使用)。

- 短期版でサンプル不足の場合はα=0扱い(長期版のみにフォールバック、`blend`の仕様)

> S_新台増台・S_移動台はこの構造を既に内包しているため対象外。S_稼働低さは店舗全体集計のため対象外。

### 店舗×日 高設定台数上限キャリブレーション(Step1実装済み・候補C)

> 2026-07にStep1(候補C)を実装。Step2(候補D・Kalman filter)は未着手、詳細は[`今後の実装予定.md`](今後の実装予定.md)参照。

「高設定割合が5割を超えることは稀」という構造的制約を店舗ごとにデータから動的推定し、Stage3出力の店舗全体集計が超過した場合にソフトに下方補正(shrinkage)する後処理。長期/短期αブレンド(直上)と同一の検出ロジック構造を応用している(`patterns.blend_scalar`。既存`blend()`は台×日の行単位Series専用のため、店舗×日の分位点集計値1本をブレンドする用途向けに新設)。

**候補比較**(データ増加での精度向上 / 月次変化への追従 / 実装コスト):

| 候補 | 概要 | 精度UP | 追従 | コスト |
|---|---|:---:|:---:|:---:|
| A. スライディングウィンドウ分位点 | 直近Mヶ月の分位点 | × | ○ | 低 |
| B. EWMA分位点 | 指数減衰で古いデータを減衰 | △ | △ | 低 |
| **C. 長期/短期αブレンド流用(Step1実装済み)** | 上記のブレンド構造をそのまま再利用 | ○ | △ | 低 |
| **D. Kalman filter(最終形・未着手)** | 真の上限を潜在状態として逐次推定 | ○ | ○ | 高 |

採用方針: 候補Dを最終形とし、単店運用中でも検証可能な候補Cを先行実装。multi_store.pyのKalman基盤構築後に候補Dへ移行。

**店舗全体での集計である点に注意**(2026-07の検討時に機種・島単位の集計と誤解した経緯があるため明記):

```python
E[高設定台数] = Σ high_prob(台)   # 機種・島単位ではなく店舗全体で集計
N = 稼働台数                      # その日の店舗全体の稼働台数
上限 = min(絶対上限0.5, α×短期分位点 + (1-α)×長期分位点 + 安全マージン)
if E[高設定台数] / N > 上限:
    offset = 二分探索でE[]/Nが上限相当に収まるようLogOddsに一律適用
```

1つの機種に投入が集中しても、店舗全体の投入率が上限内であれば発動しない(機種単位の集中投入とは無関係に、店舗全体の合計だけを見る設計)。

**実装内容(`score.py`)**:
- `compute_daily_uplimit_ratio(df, hole_name)`: 店舗×日でΣhigh_prob/稼働台数(is_invalid除外後)を集計
- `compute_uplimit(df_scored, hole_name, config)`: 長期分位点(全履歴)・短期分位点(直近M日)を`uplimit_config.json`の設定値で算出し、`patterns.blend_scalar`(α=`patterns.FIXED_ALPHA`固定。α自体の学習はStage7評価ハーネス転用後に行う方針、[`今後の実装予定.md`](今後の実装予定.md)2.2節参照)でブレンド。超過日はin-placeで`log_odds`/`high_prob`を補正
- `uplimit_config.json`: 分位点・安全マージン・絶対上限・業界一般値フォールバック(0.4)・短期ウィンドウ日数・最低必要日数・特異日除外リストを外出し(`weights.json`と同じ方針)
- データ不足(最低必要日数未満)時は業界一般値フォールバック(0.4)を使用し、offset補正はスキップ

**実データ検証結果(2026-07-06、実データ5店舗で`run_store_profile.py`実行)**:

- 発動日割合は店舗ごとに大きく異なる(3.8%〜82.7%)。特にyasuda7(153/185日=82.7%)・パーラースーパーセブン新高円寺店(114/185日=61.6%)は非常に高頻度。これは長期/短期分位点ブレンド値(0.55前後)が絶対上限0.5を上回り、`min(0.5, ...)`により実質的に絶対上限0.5固定になっているため(店舗ごとの動的な分位点が機能していない)。原因はStage3出力(high_prob)の日次平均が0.5近辺に集中している(preprocessのlog_oddsベースラインの偏り。w3=0時代に見られた狭帯域問題と同型の現象が、w3有効化後も一部店舗で残っている)
- yasuda7で台ごとの統合スコア(high_prob)コントラスト(上位10%-下位10%ギャップ)・差枚との相関を確認: コントラストは悪化なし(0.369→0.372)、Spearman相関もほぼ同水準(0.839→0.836)で90パーセンタイルキャップ試験適用時の検証結果の傾向と整合
- **要フォローアップ**: 上記の高発動率は`uplimit_config.json`の安全マージン(0.05)・絶対上限(0.5)が現在のStage3出力分布に対して厳しすぎる可能性を示唆する(発動日の割合が極端に多い場合はマージンを見直す必要がある)。詳細な対応候補は[`今後の実装予定.md`](今後の実装予定.md)2.3節参照。値の調整は運用しながら判断する暫定パラメータ

**発見した前提条件の課題(実装優先度に影響、2026-07当時)**: 回転数チャンネル(w3・`logLR_kaiten`)は`preprocess.compute_all_logLR`で常に0.0固定(Stage5学習待ちのスタブ、本ドキュメントのStage3節では「単店段階ではw3は実測相関の強さをそのまま使用」としているが未対応)。この結果、E[high_prob]/Nが常に0.5近辺の狭い帯域に収束し、上限モデルが判定対象とする「店舗全体の高設定投入率」を意味のある解像度で表現できていない可能性が高い。

**前提条件の解消(2026-07、`multi_store.py`実装により完了)**: 上記のw3有効化(Stage1b/5)を実装済み。ただし`stage3_channel_weights.json`は現時点でw3=0.0のまま(LOSOゲート未合格、または未再学習)であり、上記「実データ検証結果」で見られる狭帯域の残存はこれが一因の可能性がある。

**狭帯域問題の解消(2026-07-07、Stage3の事前確率切片β₀導入により完了)**: 上記の高発動率の根本原因は「β₀なし=暗黙にπ=0.5」のmiscalibrationだった(Stage3節「事前確率切片β₀」参照)。π=0.15導入後に全店舗(6店舗)で再実行した検証結果:

- high_prob日次平均: 0.46〜0.50 → 0.24〜0.28(中央値0.12〜0.17)に低下し、狭帯域が解消
- 上限キャリブレーション発動: yasuda7 153/185日(82.7%)→0/186日、パーラースーパーセブン新高円寺店 114/185日(61.6%)→1/186日、他店舗も0〜4日に減少(動的分位点が絶対上限0.5の下で正常に機能)
- 順位の健全性: β₀は全行一律のため店舗内順位は不変。同日実測差枚とのSpearman相関はyasuda7で0.839と導入前と同値、全店舗0.77〜0.84で維持。上位10%-下位10%ギャップは0.69〜0.77に拡大(sigmoidの飽和領域から抜けたことでコントラストはむしろ向上)
