## score.py

### 店舗×日 高設定台数上限キャリブレーション(Step1実装済み・候補C)

Stage3が台ごとに独立してsigmoid(LogOdds)を出力するため、店舗内で合計すると理論上ありえない水準まで高設定確率の期待値が積み上がることがある。`run_store_profile.py`の`run_for_hole()`内、S_稼働低さの計算後・`synthesize()`呼び出しの前で`sc.compute_uplimit(scored, hole_name)`を呼び、超過日の`log_odds`/`high_prob`をin-placeで補正してから`write_stage3_scores`を書き込む(補正後の値を機能A/Bに反映するため、この呼び出しはcompute_uplimitの後に移動している)。

- ハードキャップではなく、超過度合いに応じて連続的に縮小するソフトな正則化(ベイズ的shrinkage)。二分探索(`_solve_uplimit_offset`)で`mean(sigmoid(log_odds-offset))`が上限相当に収まるoffsetを求め、全台一律に適用する
- Stage5の階層モデル(β₀切片)に対する正則化として位置づけ。γ_storeの自由な学習は妨げない
- `patterns.py`のサブスコア(S_全台系・S_鉄板台等)は本処理より前の時点で確定済みのため影響を受けない(検出用信号とStage3出力キャリブレーションは役割が別)
- 候補比較・実データ検証結果は上記patterns.pyセクション「長期/短期αブレンド」直後を参照
- `store_profile`テーブルに`上限キャリブレーション値`・`上限信頼度`カラムを追加(`gamma_store`と同様、店舗単位のスカラー値を(ホール名,パターン)の全行に複製。既存DBへは`ALTER TABLE`でマイグレーション)
- 特異日(周年・グランドオープン等)除外は案A(`uplimit_config.json`の「特異日除外リスト」に日付文字列を手動追加。識別方法自体は引き続き未設計、初期値は空リスト)

### サブスコア一覧

| サブスコア | 系統 | データ不足時 | αブレンド |
|---|---|---|---|
| S_全台系 | 幅 | 除外 | ◯ |
| S_鉄板台 | 深さ | 履歴不足 or 欠損偏りガード抵触 → 除外 | ◯ |
| S_ローテ | 深さ | 同上 → 除外 | ◯ |
| S_新台増台 | 幅 | 該当なし → 除外 | — |
| S_移動台 | 幅 | 該当なし → 除外 | — |
| S_据え置き | 深さ | 履歴不足 → 除外 | ◯ |
| S_稼働低さ | 店舗全体 | 不明 → 除外 | — |

### 合成スコア式

```python
狙い目度 = Σ(wᵢ × Sᵢ) / Σ(wᵢ)  # 存在するサブスコアのみで再正規化
```

- データ不足のサブスコアは「0点」ではなく「計算から除外」(新規店がデータ不足だけで低評価にならないようにする)
- 初期重みは均等(wᵢ=1)。全台系×稼働低さを重め(wᵢ=1.5)にする案もあるが、実データで調整前提
- 重みは `weights.json` に外出し → コード変更なしで調整可能

### 稼働(混雑度)の推定(S_稼働低さ)

```python
# 手入力なし・スクレイピング済みの回転数から完全自動推定
推定稼働 = その店・その日の回転数集計(合計または平均)
基準値 = その店の過去の同条件(同曜日・イベント有無等)の平均
S_稼働低さ = 基準値に対する「低さ」を正規化(逆数方向)
```

> 個別台の役割②(設定の証拠としての回転数)とは目的が異なる。同じ生カラムから別の意味の指標を導出している。

### 店舗プロファイル

機能Bの「データ蓄積のみで精度向上」を実現する派生データ層。振り返り分析ごとに再計算される。

| 項目 | 内容 |
|---|---|
| 店舗ID | - |
| パターン種別 | 全台系/鉄板台/ローテ/新台増台/移動台/据え置き/稼働低さ |
| 現在のサブスコア値 | Sᵢ |
| 信頼度 | 履歴日数・サンプル数・欠損偏りフラグを反映(仮) |
| γ_store | Stage5-1で学習される店舗固有補正(複数店舗揃うまで空欄可) |
| 上限キャリブレーション値 | 店舗×日 高設定台数上限キャリブレーション(Step1)の推定値(2026-07追加、詳細は上記score.pyセクション参照) |
| 上限信頼度 | 同上の信頼度(データ量ベース) |
| 最終更新日時 | - |

- 信頼度は「学習結果」ではなく、データ量に基づく統計的信頼性の代理指標
- 出力時、内訳と信頼度を必ず併記(データが薄い新規穴場店を不当に隠さないため)

### 機種×日 全台系/高配分の判定ログ(`write_machine_judgment_log`、2026-07-09設計合意・Phase1実装済み)

`patterns.score_zentaikei_judgment`の出力(機種×日×ホール粒度)を`machine_judgment_log`テーブルへ
append-only記録する(今後の実装予定.md 1.8節)。`run_store_profile.run_for_hole`が`write_stage3_scores`
直後(上限キャリブレーション補正後のhigh_probを使うため。機能Aの表示値と揃える)に呼ぶ。

- `prediction_log`/`score_snapshot`(直近1日のみ記録)と異なり、**呼び出し側は収集済み全履歴分を
  まとめて渡す**(全台系イベントは月数回と稀なため記録を早く始める価値があり、当日完結の記述統計
  でリークの心配がないため過去分の再計算も問題ない)
- 重複ガードは事前の存在チェックではなく、`PRIMARY KEY (ホール名, 日付, 機種名)`に対する
  `INSERT OR IGNORE`で行単位に行う(初回=全履歴バックフィル、以降=新規日のみ実質追記となり、
  fase4の日次再実行でも二重記録されない。`prediction_log`のバッチ単位事前チェック方式とは異なる設計)

### 末尾版レイヤー2検定結果の保存(`write_group_calendar_conditions`、2026-07-10「末尾版」フェーズ2実装済み)

`patterns.build_group_calendar_conditions`の出力(台番号末尾グループ×日付条件のMann-Whitney U検定+
一致ルール2本、詳細は上記patterns.pyセクション参照)を`group_calendar_conditions`テーブルへ保存する。
`run_store_profile.run_for_hole`が`write_stage3_scores`直後(`write_machine_judgment_log`と同じ位置。
上限キャリブレーション補正後のhigh_probを使うため)に呼ぶ。

- `teppan_conditions`と同じ「店舗単位で全削除→再挿入」方式(収集済み全履歴を毎回再検定するため、
  `machine_judgment_log`のようなappend-only履歴蓄積は不要)
- 有意でない組み合わせも含め全条件を保存する(検出は緩く保存・使用側でゲートの方針。重複統合
  (max効果量)は保存時ではなく予測時に適用する)
- 続けて`patterns.predict_tail_group_next_day`で翌観測日の予測を計算し、予測種別`'S_末尾'`で
  `prediction_log`へ並走記録する(`_run_tail_group_predictions`。既存`write_prediction_log`の
  重複ガードをそのまま流用)

### 機種バイアス判定の保存・読込(`write_machine_bias_flags`/`read_machine_bias_list`、2026-07-14「機種バイアス除外・案A」実装済み)

`patterns.identify_machine_bias`の出力(機種名・対象店舗数・有意店舗数・有意店舗比率・バイアス判定)を
`machine_bias_flags`テーブルへ保存する(詳細はpatterns.pyセクション「機種バイアス除外・案A/案B」参照)。

- 他の店舗単位テーブルと異なり**グローバル1本**(店舗単位のキーを持たない、全店舗横断の診断情報のため)。
  `run_store_profile.run_for_hole`が店舗処理のたびに全削除→再挿入で最新化する
- バイアス判定=Falseの機種も含め全機種を保存する(「保存は緩く・使用側でゲート」の既存方針。
  診断・案B較正への入力として可視化できる形にする)
- `read_machine_bias_list`はバイアス判定=Trueの機種名リストを返す。`run_store_profile.py`の
  予測生成・`app_b.py`のおすすめ店舗スコア/有効性マトリクス/カレンダー投影の除外フィルタが使う
- `read_machine_constant_conditions`/`count_profiled_stores`は`identify_machine_bias`の入力
  (全店舗の機種×恒常条件、プロファイル済み店舗数)をDBから読み込むヘルパー
