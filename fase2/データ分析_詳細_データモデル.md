## データモデル(fase2新規追加分)

**データの流れ(2026-07改修)**: fase2の入力は`ホールデータ/turso_replica.db`(fase1が維持するTurso埋め込みレプリカ。sqlite3で読み取り専用参照)、分析成果物(`stage3_scores`/`store_profile`)の保存先は`ホールデータ/analysis.db`(ローカル専用・再計算で再生成可能)。パスと接続は`data_source.py`(`REPLICA_DB_PATH`/`ANALYSIS_DB_PATH`/`connect_replica`/`connect_analysis`/`list_holes`)に集約。旧構成の店舗別DB(`ホールデータ/{店舗名}.db`)はTurso移行前のアーカイブでありfase2は参照しない。

表示側(`app_b.py`の`_detect_pattern_periods`)は「スコア>0が連続している区間」を軽量な後処理でまとめて検出期間として可視化する(統計検定の再実行はしない近似表示、しきい値は暫定0.0)。過去に記録済みの行は書き換えない(append-only)。2026-07に[`fase4/`](../fase4/)(日次自動実行)を実装し`run_store_profile.py`は毎日自動実行されるようになったため、日付方向の粒度は日次で揃う。**2026-07-13機能B理想形(Phase6)以降、検知期間履歴での使用はS_稼働低さ(店舗全体指標で機種名を持たないため実データ切替の対象外)のみに縮小**(他パターンは`machine_judgment_log`/`stage3_scores`/`introduction_events`の機種名・台番号つき実データへ切替済み)。書き込み自体(`write_pattern_history`)は全パターン分継続。

**重複追記ガード(2026-07・fase4導入の随伴改修)**: `write_prediction_log`はINSERT前に
(ホール名, 予測種別, 使用データ最終日)の組が既存かを照会し、**該当する組のrowsを丸ごと
追記前にスキップ**する(粒度は台単位ではなくバッチ単位。1回の`run_store_profile.py`実行=
1店舗×1予測種別×1データ最終日のバッチのため十分)。データが進んでいない店舗で
`run_store_profile.py`を再実行しても、fase4のcatchup実行や手動再実行で同じ予測が
二重記録されない。DELETE/UPDATEは行わずINSERT対象を絞るだけなのでappend-only方針は維持される。

### `score_snapshot` テーブル（analysis.db、`score.write_score_snapshot`が更新。2026-07-09 タスク5「記録側先行実装」）

wᵢ(サブスコア重み)学習(今後の実装予定.md 1.6節)の教師データを**日次スナップショットとしてappend-only蓄積**する。学習側(Spearman直接最大化・店舗別ゲート・均等重みとの並走対比較)は蓄積待ちの別タスクで未実装。粒度は**店舗×使用データ最終日で1行**、値は「使用データ最終日の日次断面」の店舗平均(全期間平均ではない。教師=翌日の店舗実測差枚率に対応する説明変数にするため)。

| カラム | 型 | 内容 |
|---|---|---|
| 実行日時 | TEXT | |
| 使用データ最終日 | TEXT | PRIMARY KEY(ホール名, 使用データ最終日) |
| ホール名 | TEXT | |
| S_全台系 〜 S_稼働低さ | REAL | `SUB_SCORES`7列。使用データ最終日断面の店舗平均(NaN可) |
| 狙い目度_店舗平均 | REAL | 同断面の合成スコア店舗平均 |
| 有効重みJSON | TEXT | JSON({サブスコア列: weights×reliabilitiesの実効値}）。合成前のサブスコア別の値と実効重みを両方残すのが本質(狙い目度だけでは重みを変えた再現学習ができない) |

`run_store_profile.run_for_hole`が`synthesize`直後に、その回の`使用データ最終日`断面のみを抽出して呼ぶ。**重複ガード**は`write_prediction_log`と同型((ホール名, 使用データ最終日)が既存ならスキップ、DELETE/UPDATEなし)。タスク3で`S_据え置き`が符号付き日次値に変わった後の値であることに注意(店舗の癖としての解釈は`store_profile`の遷移列(`遷移_ベース率`/`遷移_p_stay`/`遷移_p_up`)を参照)。

`fase3/upload_analysis.py`の`TABLES`は固定6テーブルのリスト方式(`stage3_scores`/`prediction_log`/`store_profile`/`teppan_conditions`/`pattern_history`/`prediction_accuracy`)のため、`score_snapshot`は自動的にアップロード対象外(ローカル学習専用データのためこれでよい。列不整合の心配もない)。

店舗処理のたびに全削除→再挿入で最新化する。

### `ground_truth` テーブル（`ホールデータ/ground_truth.db`、正解発表の手入力データ。2026-07-12実装済み）

店舗の設定発表(LINEオプチャ/X/Webサイト等)を`fase2/ground_truth_entry.py`(ローカル専用Streamlit、
`streamlit run fase2/ground_truth_entry.py`。`app.py`には統合しない)で手入力し蓄積する。
`analysis.db`は再生成可能な成果物のため、再生成不可能な手入力データは分離して
独立ファイル(`ground_truth.db`)に保存する(バックアップはOneDrive自動同期)。
用途は1.2節(ローテ実証)・1.6節(w1/w2キャリブレーション)・1.8.1節(機種判定Phase2較正)の
教師データ(詳細は[`今後の実装予定.md`](今後の実装予定.md)3節参照)。

| カラム | 型 | 内容 |
|---|---|---|
| id | INTEGER PK AUTOINCREMENT | |
| 日付 | TEXT | |
| ホール名 | TEXT | stores.json正式名 |
| 機種名 | TEXT | ana-slo表記。NULL=店舗全体の発表 |
| 発表タイプ | TEXT | `'全456'` / `'全6'` / `'半数456'`(分母は機種単位) / `'台数指定'` / `'ローテ対象'` / `'台指定'` / `'通常日'`(弱い負例として記録) |
| 数値 | REAL | 台数(`半数456`/`台数指定`のみ使用) |
| 台番号リスト | TEXT | JSON配列文字列(`ローテ対象`/`台指定`で使用) |
| 設定値 | TEXT | 既定`'456'`。`'56'`/`'6'`/自由入力にも対応 |
| 発表タイミング | TEXT | `'事前告知'` / `'当日'` / `'事後'`(既定。発表は基本事後告知のため) |
| 媒体メモ | TEXT | 任意 |
| 入力日時 | TEXT | フォーム送信時刻の自動記録 |

**ラベルの性質**: 片側ラベル(positive-only)。発表された台=高設定は信頼できるが、発表なし=低設定では
ない(発表なしで入っている場合もある)。学習・評価は「正例が同日・同機種の未発表台よりhigh_prob上位に
来るか」のランキング型で設計する(今後の実装予定.md 3節)。

**入力フォームの絞り込み**: 選択した日付時点のレプリカ`slot_data`から「日付→ホール→機種名→台番号」の
順に候補を絞り込む(過去入力では現在の機種構成と一致しないため)。該当日がレプリカ未収集の場合は
近傍±3日の構成を候補表示にフォールバックし、自由入力欄で吸収する。

**append-only制約**: `prediction_log`と同様にDELETE/UPDATEは原則禁止。誤入力の訂正は
「フォーム内でこのセッション中に入力した直近1件のみ取消可能」に限定する(過去行の一括編集UIは無い)。

**分析DBとの分離**: `fase3/upload_analysis.py`の`TABLES`固定6テーブルには含まれず、クラウドへは
アップロードされない(ローカル資産)。
