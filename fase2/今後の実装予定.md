## 1. 機能B再設計 Phase6（将来項目）

実装済みのPhase1〜5（pattern_history・符号付きスコア・Stage7-0〜7-2・トップページ・店舗プロファイル拡張）は
`データ分析_skill.md`参照。以下はそこから意図的に切り離した将来項目。

### 1.1 Stage7-4: ウォークフォワードα学習への置き換え

`prediction_log`に長期/短期の生コンポーネント値が数十〜数百件溜まった時点で、既存の
`patterns.walk_forward_alpha`（現在未使用）をこのログに対して実行し、`FIXED_ALPHA`(=0.3)を
学習値に置き換える。

- 前提: `prediction_log`が十分な件数（目安30〜数百件）に達していること。2026-07-06時点で388〜727行程度蓄積されているため、着手判断はサンプル数と評価結果（下記1.4のprediction_accuracy）を見て行う
- 同じ評価ハーネスを、w3（回転数チャンネル）再検証（LOSO）・新規パターン採否判定・後述2節（上限モデルStep2以降）の検証にも共通利用する想定

### 1.2 Stage7-5: S_ローテ・S_据え置きの「非該当日」判定・翌日予測拡張

深さ型のうち鉄板台のみ符号付き化・翌日予測に対応済み。ローテ（`score_rotation`）・据え置き
（`score_sueki`）は現状ユニット単位で1つの定数値を全期間に適用しており、「日単位の該当/非該当」
という概念自体が実装されていないため、以下が未確定のまま残っている。

- ローテ: 「その日、集中先の台がどれであるか」を日次で判定するロジック（現状は窓全体のジニ係数1値のみ）
- 据え置き: 「連続性が切れた日」の定義（自己相関という性質上、幅型のような単純な閾値越えでは表現できない）
- 上記が確定した後、`prediction_log`の`予測種別`を`'S_ローテ'`/`'S_据え置き'`に拡張し、Stage7-1の
  `predict_next_day`と同じ枠組み（長期/短期ブレンド・append-only・リーク禁止）で翌日投影を追加する

### 1.3 Stage7-X（保留・優先度低）: 週次バックテスト再生

過去日付を人工的に区切って再計算しウォークフォワードのサンプルを増やす案。計算コストに加え、
`kaiten_bin_curves.json`等「1回学習・全店舗再利用」ファイルの過去時点バージョンとの整合が
複雑になるため保留。Stage7-4（実運用データの自然蓄積によるα学習）で収束が遅すぎると
判断された場合のみ検討する。

### 1.4 未確定・実装時に詰める項目（機能B再設計から持ち越し）

1. トップページの機種名/台番号レベルの候補出しロジック: 既存`load_hot_machines`のように
   high_prob上位を機械的に拾うだけか、パターン別の理由文を紐付けて生成するかは未確定
   （現状は前者の最小構成で実装済み。後者への拡張余地）
2. 深さ型「非該当日判定」の具体的な閾値・ロジック（1.2節と同じ論点。ローテ・据え置きそれぞれで個別設計が必要）
3. 符号付きスコア導入に伴う`app_a.py`側の表示レンジ見直し範囲: `app_b.py`は`range_x=[-1,1]`・
   `RdBu`配色へ対応済みだが、`app_a.py`の「統合スコア(0〜6)」はhigh_prob由来の別指標のため
   対応不要と判断済み。ただし将来`app_a.py`側で符号付きサブスコアを直接表示する要望が出た場合は
   再検討が必要
4. S_ローテ・S_据え置きの「継続する」という前提の具体的な定式化（1.2節のStage7-5着手時に必要）

## 2. 店舗高設定上限モデル Step2以降（将来項目）

Step1（候補C・長期/短期αブレンド流用）は`データ分析_skill.md`「店舗×日 高設定台数上限
キャリブレーション」に実装済みとして記載済み。以下はそこから意図的に切り離した将来項目。

### 2.1 Step2: Kalman filterへの移行（候補D、最終形）

```python
状態_t = 状態_{t-1} + プロセスノイズ   # 月次変化を緩やかなドリフトとしてモデル化
観測_t = 状態_t + 観測ノイズ           # その日の高設定割合
Kalman更新:
  不確実性が高い(データ少) → ゲイン大 → 新しい観測を強く反映
  不確実性が低い(データ多) → ゲイン小 → ノイズに惑わされにくい
上限 = min(0.5, 状態推定値 + 安全マージン)
```

- 前提: `multi_store.py`側で店舗パラメータ用のKalman filter基盤が実際に構築されること
  （現時点で`multi_store.py`にKalman機構はまだ存在しない。γ_storeはLOSO交差検証によるスカラー
  補正のみ）。基盤ができた時点で上限推定ロジックをそちらに統合し、候補C（Step1）を置き換える
- 採用理由（候補比較時の結論）: 「データ増加で精度向上」と「時間変化への追従」を単一パラメータ
  （プロセスノイズ分散）で連続的に両立できる点が候補A/B/Cにない特性のため。候補比較の詳細は
  `データ分析_skill.md`「店舗×日 高設定台数上限キャリブレーション」参照

### 2.2 α自体の学習（候補Cの固定値からの置き換え）

Step1は指示通り`patterns.FIXED_ALPHA`(=0.3)の固定値で暫定実装した。本来の設計では、
1節のStage7評価ハーネス（`prediction_log`/`evaluate_predictions.py`）を店舗×日集計向けに
転用し、「上限推定値ベースの補正が実測差枚の予測力を上げるか」をSpearman相関で検証してから
αを決定する想定だった。

- Stage7は台×日単位（S_鉄板台の翌日予測）の評価だが、本項目は店舗×日単位（E[高設定台数]/N）
  の評価であり、集計粒度が異なる。ただし「長期版・短期版をαブレンドし、ウォークフォワードで
  実測に対する予測力を検証する」という骨格自体は共通のため、`prediction_log`のテーブル構造・
  `evaluate_predictions.py`の評価ロジックを店舗×日集計向けに転用できる（二重実装を避けるため
  新規の評価スクリプトは作らない）
- 着手前提: 1.1節（Stage7-4）と同様、評価ハーネス側の運用実績が十分に溜まってから行う

### 2.3 発動日割合の偏り（要フォローアップ・実データで確認済みの課題）

2026-07-06の実データ検証（5店舗）で、上限キャリブレーションの発動日割合が店舗により
3.8%〜82.7%と大きくばらつき、一部店舗（yasuda7: 82.7%、パーラースーパーセブン新高円寺店: 61.6%）
で非常に高頻度という結果が出た。

- 原因: 長期/短期分位点ブレンド値が絶対上限0.5を上回り、`min(0.5, ...)`により実質的に絶対上限
  0.5固定になっている（店舗ごとの動的推定が機能していない）。Stage3出力(high_prob)の日次平均が
  0.5近辺に集中していることが根本原因で、w3=0時代に見られた狭帯域問題と同型の現象
- 対応候補（未決定）:
  - `uplimit_config.json`の安全マージン(0.05)・絶対上限(0.5)の見直し
  - Stage3出力の分布そのもの（high_probが0.5近辺に集中する傾向）の是正が先か、上限モデル側の
    パラメータ調整で吸収するかの判断
  - `stage3_channel_weights.json`のw3が引き続き0.0（LOSOゲート未合格）である影響の切り分け
- 判断材料が揃うまでは、`uplimit_config.json`の値は運用しながら調整する暫定値のまま据え置く

### 2.4 特異日（周年・グランドオープン等）の自動識別（案B、保留）

Step1では案A（`uplimit_config.json`の「特異日除外リスト」への手動追加、初期値は空）で暫定運用中。
案B（極端な乖離自体を「特異日候補」として別枠検出し、この制約をスキップする）は実装コスト中・
誤判定リスクありのため、案Aの運用で問題（本物の強い日が機械的に潰される事例）が顕在化してから
検討する。

### 2.5 日タイプ別（イベント日/通常営業日）の上限分割

上限を店舗単位で1本にするか、日タイプ別（イベント日/通常営業日）に分けて学習するかは未決定。
Step1実装時点ではデータ量を見て判断するとしていたが、実データでの判断はまだ行っていない。
2.3節の発動日割合の偏りの原因調査と合わせて検討するのが効率的。

### 2.6 Tier A機種によるマクロ整合性クロスチェック

Tier A機種が十分にある店舗については、マクロ整合性逆算（`multi_store.check_macro_consistency`と
同型の手法、Stage6実装済み）を上限推定のクロスチェックとして併用できないか、Phase F（複数店舗、
実装済み）の枠組みを流用して再検討できる。優先度は低い（未着手）。
