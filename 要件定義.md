# 要件定義

## 目的
パチスロホールのデータを収集・分析し、高設定が入る台を予測して、iPhoneから直接Webで確認できるシステムを構築する。

## 機能要件

### 1. データ収集（実装済み）— [`fase1/`](fase1/)
> 詳細仕様: [`fase1/データ収集_skill.md`](fase1/データ収集_skill.md)

- ana-slo.com から指定ホール・期間のスロットデータをスクレイピング
- 収集済みデータを SQLite に保存（店舗ごとにファイル分離・重複スキップ）
- 不正なURL入力時は再入力を促す

### 2. 設定推測・パターン分析（実装済み）— `fase2/`
> 詳細仕様: [`fase2/データ分析_skill.md`](fase2/データ分析_skill.md) / 構成図: [`fase2/データ分析_構成図.md`](fase2/データ分析_構成図.md)

#### 基盤レイヤー: 個別台 設定推測アルゴリズム（Stage 0〜6）
- **Stage 0**: TEXT→数値変換、台×日のユニーク化
- **Stage 1a**: 機種別Tier判定（A/B/C）と回転数の機械的バイアス補正（機種×データ提供元単位で共通・全店舗再利用可）
- **Stage 1b**: 回転数チャンネルの行動的成分を店舗ごとに学習（複数店舗データが必要。`fase2/multi_store.py`で実装済み。全店舗データをプールして機種別デシルカーブを学習し、`kaiten_bin_curves.json`に保存・単店実行時も自動再利用）
- **Stage 2**: 3チャンネルのlogLR計算 — ①RNG確率（ポアソン尤度）②差枚（正規分布、`machine_setting_specs.json`の理論機械割(payout)があれば優先使用・無ければ経験的分位点にフォールバック）③回転数行動（経験的非線形logLR。2026-07にStage1b実装により有効化済み。未学習時は0.0にフォールバック）
- **Stage 3**: 3チャンネルを重み付き合算した LogOdds → sigmoid で高設定確率（0〜1）。**この出力がパターン検出の入力データ**。チャンネル重み(w1/w2/w3)はStage5学習後`stage3_channel_weights.json`から自動読込
- **Stage 4**: 回転数不足台の判定不能除外 + 欠損偏りガード（判定不能率差が閾値超過時に信頼度低下またはスキップ）
- **Stage 5**: 複数店舗データでの階層モデル精緻化（店舗固有補正 γ_store を自動学習。`fase2/multi_store.py`で実装済み。RNGベースlogLR(y1)を疑似正解とした店舗別回帰でγ_store・チャンネル重みβ3を学習し`store_profile.gamma_store`・`stage3_channel_weights.json`に保存）
- **Stage 6**: 教師ラベルなし検証戦略（Tier再現性・マクロ整合性・店舗群クロスバリデーション。`fase2/multi_store.py`の`check_tier_reproducibility`/`check_macro_consistency`で実装済み）

#### パターン検出（幅型 / 深さ型）
- **幅(breadth)型**（当日データ・少ないデータでも検出可）
  - S_全台系: 同日・同機種/列/島の横断スコア集計
  - S_新台増台: 増台後の直近Stage3スコア移動平均 − 基準値（自動フェードアウト）
  - S_移動台: 移動後の直近Stage3スコア移動平均 − 店舗内同機種平均（移動判定ロジックのトリガー後）
- **深さ(depth)型**（縦の履歴が必要・通うほど精度向上）
  - S_鉄板台: ACF→PDM（→Lomb-Scargle）周期探索 + 既知カレンダー17候補×FDR補正×効果量ゲート
  - S_ローテ: 集中度指標（エントロピー/ジニ係数）+ 並べ替え検定 + FDR補正 + 効果量ゲート
  - S_据え置き: Stage3スコアの自己相関
- **S_稼働低さ**: 店舗全体の回転数集計から混雑度を自動推定（手入力なし）
- **(検討中・保留) 店舗×日 高設定台数上限キャリブレーション**: Stage3出力(高設定確率)の店舗全体集計が「高設定割合5割超は稀」という構造的制約を超えた場合に下方補正するモデル。詳細は[`fase2/データ分析_skill.md`](fase2/データ分析_skill.md)・[`fase2/追加検討 店舗高設定上限モデル.md`](fase2/追加検討%20店舗高設定上限モデル.md)参照。2026-07の実データ検証でコントラストへの悪影響は無いことを確認。実装見送りの前提条件だった回転数チャンネル(w3)未有効化は2026-07にStage1b/5(`multi_store.py`)実装により解消済み。本キャリブレーション自体(Phase G)は引き続き未着手・検討中

#### 長期/短期ブレンド（v0.6新規）
- S_全台系・S_鉄板台・S_ローテ・S_据え置きについて、同一検出ロジックで長期版（全履歴）と短期版（直近Mウィンドウ）を並行算出
- `ブレンド = α×短期版 + (1-α)×長期版`。αは店舗×サブスコア種別ごとにウォークフォワード検証で学習

#### スコア統合
- `狙い目度 = Σ(wᵢ×Sᵢ) ÷ Σ(wᵢ)` — データ不足のサブスコアは「0点」ではなく「除外」して再正規化
- 重みは `weights.json` に外出し（コード変更なしで調整可）

#### 出力3系統
- **機能A**: 1店舗内の台を手動で見比べる可視化ツール（Streamlit）。手動フィルタ（期間/機種/台番号/推定設定レンジ/列/島）
- **機能B-詳細**: 蓄積データからのパターン検出結果・店舗プロファイルを全項目表示するダッシュボード
- **機能B-簡潔**: 次回来店前に見る狙い目メモ（結論+最低限の根拠のみ）

#### （検討中・未実装）機能B再設計
2026-07に再設計方針を議論・確定（実装未着手）。詳細は[`fase2/追加検討_機能B再設計.md`](fase2/追加検討_機能B再設計.md)参照。
- トップページ: 当日・翌日の狙い目ランキング（強い店・弱い店を1軸で予測、既知カレンダー条件+未知周期の両方を候補表示、信頼度・的中率併記）
- 店舗プロファイル: パターン別の強さ・弱さ表示（幅型は符号付きスコア、深さ型は該当日/非該当日判定）・検知期間履歴・当月カレンダーヒートマップ（統合スコア+パターン別）
- 将来: 予測ログと実測データを突き合わせて予測精度を自己検証する学習ループ（Stage7、ウォークフォワード必須）

### 3. 配信・公開（iPhoneからのWebアクセス）（未実装）— `fase3/`
LINEへのプッシュ通知は廃止し、fase2の出力（機能A/B）をiPhoneから直接Webで閲覧できる形に置き換える。fase3は分析ロジックそのものではなく、「fase2の成果物をどう届けるか（公開・デプロイ・アクセス制御）」を担う層と位置づける。機能A/Bのコード自体はfase2内の各モジュール（score.py等）に依存するため、fase2フォルダに残す。

#### 移行方針（段階的にクラウド化）
1. **DB移行**: 店舗ごとのSQLiteファイルから、クラウドDB「**Turso**」（libSQL/SQLite互換）へ移行することを決定。無料枠が広い（5GBストレージ／月5億行読み取り／月1000万行書き込み）ことと、SQLite方言互換でfase1の書き換えコストが小さいことが決め手（Supabase・Turso共にGitHub Student Developer Packの公式特典対象外であることは確認済み）
2. **Stage A（現段階で着手）**: fase1のスクレイピングのみをクラウド上で実行し、Turso DBへ書き込む。fase2の分析・Webアプリ（機能A/B）は引き続きPC上で実行し、同じTurso DBを参照する
   - **実行環境**: GitHub Actionsの定期実行（`schedule: cron`）。Selenium等のブラウザ操作が不要な素の`requests`実装のため、追加費用なく無料枠内で運用可能
   - **店舗URL一覧の管理**: リポジトリ内の`stores.json`で管理。店舗数・蓄積データ量は今後も増加予定のため、Git管理で追加・削除の履歴を追える形を採用（GitHub Actionsの変数管理やTurso内テーブル化は現段階では見送り）
   - **`メイン.py`の非対話化**: 現状の`input()`による対話入力（開始日・終了日・店舗URL）を廃止し、`stores.json`読み込み＋「前回取得済み日の翌日〜当日」の自動算出に変更。無人実行（GitHub Actions）に対応するための必須変更
   - **PC上での手動実行は引き続き可能**: クラウド化は「DBの置き場所」と「定期実行の主体」が変わるだけで、既存のPC上での手動実行を置き換えるものではない
   - **容量の監視**: 店舗数・データ蓄積に伴いTurso無料枠（ストレージ5GB）が将来逼迫する可能性があるため、運用しながら使用量を確認し、必要に応じて有料プラン（Developer: $4.99〜/月、9GBストレージ）への移行を検討する
3. **Stage B（将来）**: fase2のWebアプリ（機能A/B）自体もクラウドホスティング（Streamlit Community Cloud等）へ移行し、PCの起動状態に依存せずiPhoneから直接アクセスできる状態にする

#### Stage Bまでの暫定手段（検討中）
- PCを起動した状態のまま、Tailscale等のVPN/トンネル経由でiPhoneから機能A/Bへ直接アクセスする運用も選択肢としてありうる

### 4. 日次自動実行（未実装）— `fase4/`
- 日次自動スケジューリング

## 非機能要件
- 段階的にクラウド移行（現状: PC上で完結 → Stage A: DB+スクレイピングのみクラウド化 → Stage B: 分析Webアプリも含め全てクラウド化）
- スクレイピングは礼儀正しく（リトライ＋待機）
- 手入力なし（稼働推定含め全て自動収集・自動導出）
