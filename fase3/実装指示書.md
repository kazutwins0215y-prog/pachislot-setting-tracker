# fase3: 配信・公開（Stage B）実装指示書

設計の背景・全体像は [`fase3/配信公開_設計.md`](配信公開_設計.md) を先に読むこと。
本書はそれを実装タスクに分解した指示書。**タスク0→5の順に実装する**
（タスク0〜2はPC側のみで完結し単体でも価値がある＝クラウドバックアップ効果。先行導入可）。

## 前提知識（実装セッションが最初に確認すべきコード）

| ファイル | 確認ポイント |

| `fase2/evaluate_predictions.py` | `prediction_log`に対してUPDATEを行うか（append-onlyか）。**UPDATEがある場合はタスク1のprediction_log転送方式を要調整**（IDウォーターマーク+直近対象日の再upsertに変更） |

## タスク0: 分析用Turso DBの新設（手動セットアップ）

**目的**: 生データDBと物理分離された分析専用DBを用意する（設計書「案Cを選んだ理由」参照）。

**手順**（Tursoダッシュボードまたは`turso` CLI。ユーザーと一緒に実施）:
1. 既存DBと同じorganization・**同じロケーション（Tokyo/nrt）**に新DBを作成
   （名前例: `pachislot-analysis`）
2. 認証トークンを発行し、`.env`に2行追記:
   ```
   TURSO_ANALYSIS_DATABASE_URL=libsql://pachislot-analysis-....turso.io
   TURSO_ANALYSIS_AUTH_TOKEN=...
   ```
3. `.env`がGit管理外であることを確認（既存の`.gitignore`設定のまま）

**検証**: `turso db list`（またはダッシュボード）で新DBが見えること。

## タスク1: `fase3/upload_analysis.py` — 分析成果物の差分upsert（新規）

**目的**: `ホールデータ/analysis.db`の6テーブルを分析用Tursoへ転送し、クラウドアプリが
読める状態にする。fase4から毎日呼ばれるほか、手動実行も可能。

**CLI**:
```
py -3.12 fase3/upload_analysis.py           # 差分upsert（日次用）
py -3.12 fase3/upload_analysis.py --full    # 全再構築（アルゴリズム変更後に手動実行）
```

**仕様**:
- **接続ヘルパー** `fase3/analysis_turso.py` を新設し、`fase1/db.py`の`get_connection()`と
  同じ埋め込みレプリカ方式で分析DBへ接続する:
  - 環境変数: `TURSO_ANALYSIS_DATABASE_URL` / `TURSO_ANALYSIS_AUTH_TOKEN`（`.env`から読む。
    dotenvの読み込みパスは`fase1/db.py`と同じくプロジェクトルート）
  - レプリカファイルのパスは引数で受け取る（PC側=`ホールデータ/turso_analysis_replica.db`、
    クラウド側=`ANALYSIS_DB_PATH`。タスク3で同じヘルパーを使い回すため）
  - libsql接続コードはこのヘルパーに閉じ込め、upload_analysis.py/bootstrap.pyには書かない
- ローカルの`analysis.db`（`run_store_profile.py`の出力＝PC側の正）は`sqlite3`で
  読み取り専用接続（`mode=ro`）
- リモート側に分析6テーブルが無ければ`analysis.db`の`sqlite_master`からCREATE文を
  取得して作成する（スキーマをコード内に重複記述しない）
- **転送方式**（設計書の表のとおり）:
  - `stage3_scores`: リモートの`MAX(日付)`（接続時sync済みのローカルレプリカから読む＝
    読み取り課金なし）を基準に、`日付 > (MAX(日付) - マージン日数)` の行を
    `INSERT OR REPLACE`。マージンは冒頭定数 `REUPSERT_MARGIN_DAYS = 7`
  - `prediction_log`: リモートの`MAX(予測ID)`より大きいIDの行を、**予測IDを明示して**INSERT
    （前提知識欄のevaluate_predictions.py確認結果によっては直近対象日の再upsertを追加）
  - `store_profile` / `teppan_conditions` / `pattern_history` / `prediction_accuracy`:
    DELETE全行→全INSERT（全置換）
  - リモートが空（初回）はウォーターマークがNULL→自然に全件アップロードになることを利用する
    （初回専用コードを書かない）
- **`--full`**: 6テーブルをDROP→再作成→全件INSERT。接続先が分析DBなので生データには
  構造的に触れられないが、DROP対象のテーブル名は6テーブルの明示リストに限定すること
- コミットは5,000行単位のバッチ。最後に`con.sync()`でローカルレプリカも最新化
- 終了時サマリをprint: テーブルごとの転送行数・所要時間・使用モード
- 転送行数の合計が異常に大きい場合（例: 差分モードで10万行超）はWARNINGを出す
  （ウォーターマーク不整合の検知。処理は継続してよい）

**検証**:
1. `--full`を1回実行し、リモート行数がローカル`analysis.db`と全テーブル一致すること
   （検証クエリはsync後のローカルレプリカ`turso_analysis_replica.db`に対して実行）
2. 続けて差分モードを実行 → stage3_scoresはマージン分のみ、prediction_logは0行、
   小テーブルは全置換、が転送されること
3. `run_store_profile.py --hole <1店舗>`で1店舗だけ更新→差分モード実行→
   その店舗の新規日分だけが転送されること

## タスク2: `fase4/run_daily.py` — アップロードステップ追加

**変更点**: `run_evaluate_and_profile()`の後（サマリ出力の前）に
`fase3/upload_analysis.py`（差分モード）のサブプロセス実行を追加する。

- コマンドは`PY_FASE1_CMD`（=`py -3.12`）を流用、cwd=`fase3/`
- 既存の`run_subprocess()`/`RunStats`の枠組みに乗せ、サマリにステップとして出す
- **失敗しても異常終了させない**（ERRORログのみ）。翌日の差分実行が自動的に追いつくため
  （設計書「自己修復性」）。403時は既存フローどおり評価・分析ごとスキップされるので
  アップロードも実行されない（変更不要）

**検証**:
1. `py -3.12 run_daily.py --mode catchup`を実行し、収集→評価→分析→アップロードが
   直列に走りサマリに4ステップ出ること
2. `upload_analysis.py`を一時的に失敗させ（例: 引数エラー）、run_daily自体は
   正常終了しERRORログが残ること

## タスク3: クラウド起動用bootstrap + エントリポイント（新規）

**配置**: `fase3/streamlit_app.py`（デプロイ時のメインファイル）と
`fase3/bootstrap.py`（データ実体化ロジック。streamlit_appからimport）。

**bootstrap.pyの仕様**:
- 認証情報4つを環境変数またはStreamlit Secretsから取得
  （`st.secrets`にあれば環境変数へ転写してから各接続ヘルパーを使う）
- ①生データDBをsync → `REPLICA_DB_PATH`（`fase1/db.py`の`get_connection()`を利用）
  ②分析DBをsync → **`ANALYSIS_DB_PATH`に直接**（タスク1の`analysis_turso.py`ヘルパーに
  このパスを渡す。レプリカはSQLite互換なので抽出・コピー処理は不要）
- パスは`fase2/data_source.py`の`REPLICA_DB_PATH`/`ANALYSIS_DB_PATH`をimportして使うこと
  （fase2の解決結果と必ず一致させる）

**streamlit_app.pyの仕様**:
- `sys.path`に`fase1/`・`fase2/`・`fase3/`を追加→bootstrap実行→`import app`（fase2のapp.py）の順
- **制約**: `app.py`がst.set_page_config()を呼ぶため、bootstrapは
  `@st.cache_resource(ttl=3600, show_spinner=False)`で包み、st系のUI要素を一切出さないこと
- サイドバー等に「データ再同期」ボタンを追加する場合は、set_page_config後
  （=app import後）に出す仕組みにするか、Streamlitの制約上難しければ
  TTL(60分)のみで初版とする（ボタンは必須ではない）

**検証（ローカルで可能）**:
1. `ホールデータ/turso_replica.db`と`analysis.db`を**一時的に退避**（削除はしない。
   別名リネームで対応）した状態で`streamlit run fase3/streamlit_app.py`を実行し、
   bootstrap経由で2ファイルが再生成され、トップページ・機能A・機能B-詳細が
   ローカル実行時と同じ内容で表示されること
2. 確認後、退避したファイルを元に戻す（**analysis.dbはPC側では`run_store_profile.py`の
   出力が正のため必ず元に戻す**。bootstrapが作ったanalysis.db=レプリカのままにすると
   翌日のrun_store_profileがレプリカファイルに直接書き込んでしまう）

## タスク4: Streamlit Community Cloudへのデプロイ

**事前準備**:
- リポジトリルートに`requirements.txt`を新設。内容はfase2/fase3の全importを走査して
  確定する（streamlit / pandas / numpy / libsql / python-dotenv ほか、
  scipy・plotly等はfase2の実際のimportに従う）。`fase1/requirements.txt`は触らない
- fase3の新規ファイル一式をコミット・push（**pushはユーザーの指示を得てから**）

**デプロイ手順**（Streamlit Community Cloud側の操作）:
1. https://share.streamlit.io にGitHubアカウントでログインし、
   `pachislot-setting-tracker`リポジトリを接続
2. メインファイル=`fase3/streamlit_app.py`、**Advanced settingsでPython 3.12を指定**
3. Secretsに4つ登録: `TURSO_DATABASE_URL` / `TURSO_AUTH_TOKEN` /
   `TURSO_ANALYSIS_DATABASE_URL` / `TURSO_ANALYSIS_AUTH_TOKEN`
4. アプリを**非公開（Private）**に設定し、viewerに自分のGmailを招待
5. デプロイ後の確認: (a) コールドスタートが完走するか（libsqlのwheel解決を含む）
   (b) メモリ1GB内に収まるか (c) iPhoneのSafariでログイン→3画面の表示・操作

**うまくいかない場合の代替**（この順に検討。実装前にユーザーへ相談）:
- libsqlがインストール不可 → bootstrapのsyncをlibsql-client（HTTP経由）でのDB取得に差し替え
- メモリ不足 → Hugging Face Spaces（無料CPU、16GBストレージ）へ切り替え

## 実装上の注意（プロジェクト共通ルール）

- 返答・コメント・ドキュメントは日本語
- ファイルの削除・外部公開はしない。特にタスク3検証でのDBファイル退避は**リネームのみ**
  （削除禁止）。`prediction_log`はリモート側でもappend-only厳守（`--full`のDROPは
  分析DBの6テーブル明示リストのみ・生データDBには接続しない）
- コード実装に着手する前にアプローチをユーザーに説明し承認を得る
  （本指示書の提示＝設計承認済みだが、指示書から逸脱する場合は再確認）
- コミット・push・デプロイ（外部公開に類する操作）はユーザーの指示があった場合のみ
- 分析用Tursoへの`--full`実行は検証目的でも1〜2回に留める（書き込み枠の浪費防止）
