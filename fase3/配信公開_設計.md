# fase3: 配信・公開（Stage B）設計書

fase2の出力（機能A/B）をiPhoneから直接Webで閲覧できるようにする層の設計。
要件定義「3. 配信・公開」のStage B（クラウドホスティング）を実現する。
実装タスクへの分解は [`fase3/実装指示書.md`](実装指示書.md) を参照。

## 決定事項（2026-07-07 議論）

| 論点 | 決定 | 根拠 |
|---|---|---|
| 分析成果物の連携方式 | **案C: 分析専用のTurso DBを新設**（生データDBと分離）し、fase4の最後に差分upsert | 下記「案Cを選んだ理由」参照。当初案Aで設計したが同日中に案Cへ変更 |
| つなぎ運用（Tailscale） | **省略**し、Stage B本命を直接構築 | 即時閲覧の必要なし（ユーザー確認済み）。アプリは読み取り専用と判明し構築リスクが小さい |
| ホスティング先 | **Streamlit Community Cloud**（非公開アプリ+viewer招待） | 完全無料・Streamlitネイティブ・リポジトリは既にGitHubにある |
| 料金 | **0円**で構築・運用可能 | 下記「料金と無料枠」参照。Turso無料枠はorganization全体での集計のためDBを2つに分けても枠は変わらないが、超過もしない |
| アルゴリズム変更時 | `--full`モードで**分析DBのみ全再構築**（約80万行書き込み/回） | 生データDBと物理分離されているため、生データを壊すリスクなしに気兼ねなく実行できる |

### 案Cを選んだ理由（案Aからの変更・2026-07-07）

1. **開発段階でアルゴリズム修正を短期的に繰り返す見込み**のため、`--full`全再構築を
   頻繁に実行する。分析DBが独立していれば、DROP対象のDBに生データが存在せず
   操作ミスの影響範囲が構造的にゼロになる
2. **新店舗追加でデータが数倍に増える見込み**のため、fase1が毎日syncする生データDBに
   分析テーブルの容量・行数を混ぜない（生データレプリカのsync時間・サイズを純粋に保つ）
3. 副次的な簡素化: クラウド側は分析DBのレプリカsync結果を**そのまま`analysis.db`の
   パスに置くだけ**でよく、案Aで必要だった「レプリカから分析テーブルを抽出する処理」が
   丸ごと不要になる

### 設計を単純化した重要な事実

- **Webアプリ（機能A/B/トップ）は完全に読み取り専用**。「狙い目メモ」は保存済みデータから生成するテキストであり、ユーザー入力の書き込みは存在しない（`app_b.py`確認済み）。クラウド→Tursoの書き戻し経路は不要。
- **データの鮮度は今後もPC（fase4）依存**。スクレイピングは403問題で住宅用IPに縛られるため、fase3が変えるのは「閲覧可能性」のみ。PCが数日止まればデータは古くなる（割り切り）。

## 全体構成

```
[PC・毎日] fase4/run_daily.py（既存）
  収集(fase1) → 評価(evaluate_predictions) → 分析(run_store_profile)
  → ★新設: fase3/upload_analysis.py が分析成果物を分析用Tursoへ差分upsert
        │
        ├─ [Turso: 生データDB]（既存・fase1が書く）
        │    slot_data / missing_data
        │
        └─ [Turso: 分析DB]（★新設・fase3が書く）
             stage3_scores / store_profile / teppan_conditions /
             pattern_history / prediction_log / prediction_accuracy
        │ それぞれ埋め込みレプリカsync（起動時＋TTL）
        ▼
[Streamlit Community Cloud] fase3/streamlit_app.py（★新設エントリポイント）
  bootstrap: ①生データDBをsync → ホールデータ/turso_replica.db
             ②分析DBをsync    → ホールデータ/analysis.db（パスに直接配置）
             ③fase2/app.py をそのまま起動（fase2は無改修）
  認証: 非公開アプリ + viewerメール招待（Googleログイン）
        ▼
[iPhone] Safariからアクセス（PCの起動状態と無関係に24時間閲覧可）
```

### fase2を無改修に保つ仕組み

`fase2/data_source.py` は `ホールデータ/turso_replica.db`（読み取り専用）と
`ホールデータ/analysis.db` の2ファイルを固定パスで読む。クラウド側のbootstrapが
この2ファイルを**同じ相対パスに実体化**することで、fase2のコードは1行も変えずに動かす:

1. `turso_replica.db` — 生データDBのlibsql syncで生成（PC側と同じ仕組み）
2. `analysis.db` — 分析DBのlibsql syncで生成し、**そのままこのパスに置く**
   （レプリカはSQLite互換ファイルなので、fase2からは通常のanalysis.dbと区別がつかない。
   抽出・コピー処理は不要）

「fase3は届ける層・分析ロジックはfase2」という要件定義の位置づけをコードでも維持する。

## データ連携設計（PC→分析用Turso 差分upsert）

`fase3/upload_analysis.py`。分析DBへの接続は`fase1/db.py`の`get_connection()`と同じ
埋め込みレプリカ方式で、fase3側に小さな接続ヘルパーを新設する（環境変数
`TURSO_ANALYSIS_DATABASE_URL`/`TURSO_ANALYSIS_AUTH_TOKEN`、PC側レプリカファイルは
`ホールデータ/turso_analysis_replica.db`）。接続時のsyncによりリモート側ウォーターマークの
読み取りはローカルレプリカで完結する（読み取り課金なし）。ローカルの`analysis.db`
（`run_store_profile.py`の出力＝PC側の正）は`sqlite3`で読み取り専用接続する。

| テーブル | 行数(現在) | 転送方式 | 根拠 |
|---|---|---|---|
| `stage3_scores` | 約39.6万 | **日付ウォーターマーク** + 直近マージン再upsert（既定7日） | 台×日の全履歴で最大。新規日分（約3,000行/日）+直近再計算分のみ転送 |
| `prediction_log` | 約3,800 | **予測IDウォーターマーク**（append-only・IDを明示コピー） | AUTOINCREMENT PKで単調増加。将来10万行超に成長するため全置換は不適 |
| `store_profile` | 42 | 全置換（DELETE→INSERT） | 小テーブル。毎回数十〜数百行で転送コスト無視できる |
| `teppan_conditions` | 478 | 全置換 | 同上 |
| `pattern_history` | 210 | 全置換 | 同上 |
| `prediction_accuracy` | 5 | 全置換 | 同上 |

- **初回投入の特別処理は不要**: リモートが空ならウォーターマークがNULL→自動的に全件アップロードになる
- **`--full`モード**: 分析DBの6テーブルをDROP→再作成→全件INSERT。アルゴリズム変更で
  過去日を再計算した場合に手動実行する（約80万行/回）。接続先が分析DBなので
  生データ（`slot_data`/`missing_data`）には構造的に触れられない
- **書き込み方式（実装時に判明した重要な変更）**: `cur.executemany()`は埋め込みレプリカ
  経由だと1行ごとに個別のネットワーク往復が発生し致命的に遅い（実測: 約0.12秒/行。
  80万行では十数時間かかる計算になり、初回実装では実際にハングした）。そのため
  `SQL_CHUNK_SIZE`（既定1000行）ごとに複数行分の`VALUES (...), (...), ...`を1本の
  INSERT文にまとめて`execute()`する方式に変更した（SQLiteの1文あたり変数上限
  約32766を、最大列数13でも十分下回るチャンク幅）。この方式で実データ約40万行の
  `--full`が約5分で完了することを確認済み
- **自己修復性**: アップロード失敗日があっても、翌日の実行がウォーターマーク差分で自動的に追いつく

### 書き込み行数の見積り

| 項目 | 行数/月 |
|---|---|
| 日次差分（新規3,000行+マージン7日分 約21,000行 ≒ 24,000行/日） | 約72万 |
| 小テーブル全置換（約700行×2回/日） | 約4万 |
| **合計（通常運用）** | **約76万行/月（無料枠1,000万の約8%）** |
| `--full`全再構築（臨時） | 約80万行/回 |

※新店舗追加でデータがN倍になれば上記もほぼN倍になる。5倍でも月380万行+`--full`400万行/回で
枠内だが、店舗数が2桁に乗る頃に`--full`の実行頻度と合わせて再見積りすること。

### PC側に増えるファイル

- `ホールデータ/turso_analysis_replica.db`（約80MB・upload_analysis.pyが維持する
  分析用Tursoの埋め込みレプリカ。ウォーターマーク読み取り専用で、消えても次回syncで再生成される）
- 生データレプリカ（`turso_replica.db`）には分析テーブルが混入しない（案Aとの違い）

## クラウド側設計（Streamlit Community Cloud）

- **エントリポイント**: `fase3/streamlit_app.py`。デプロイ設定でこのパスを指定
- **bootstrap制約**: `fase2/app.py`がモジュールレベルで`st.set_page_config()`を呼ぶため、
  bootstrapは**st系UIを一切出さずに**（`@st.cache_resource(show_spinner=False)`等）
  2つのsyncを済ませてから`app`をimportする
- **鮮度管理**: データ更新は1日1回（朝）なので、bootstrapのキャッシュTTLは60分+
  サイドバーに手動「データ再同期」ボタン（キャッシュクリア→rerun）
- **コールドスタート**: スリープ復帰時に約200MB（生データ+分析）のsyncが走り
  数十秒〜1分程度かかる。個人利用・1日数回の閲覧では許容（Streamlit Cloud無料枠の仕様）
- **Python 3.12指定**: デプロイ時のAdvanced settingsで指定（libsqlはLinux+3.12のwheelあり。
  ローカルWindowsの3.14ビルド問題はクラウドでは発生しない）
- **依存関係**: リポジトリルートに`requirements.txt`を新設（Streamlit Cloudが参照する場所）

## 認証・セキュリティ

- リポジトリは非公開（既存）→ アプリも**非公開アプリ**としてデプロイ（無料枠で1個まで可）
- 閲覧はviewer招待制: 自分のGoogleアカウント（Gmail）を許可リストに登録し、
  iPhoneのSafariでGoogleログインして閲覧
- Secretsは4つ: `TURSO_DATABASE_URL` / `TURSO_AUTH_TOKEN`（生データDB・既存）と
  `TURSO_ANALYSIS_DATABASE_URL` / `TURSO_ANALYSIS_AUTH_TOKEN`（分析DB・新規）を
  Streamlit CloudのSecretsに登録（リポジトリにトークンを含めない。`.env`は従来どおりGit管理外）

## 実装時に判明した追加事項（2026-07-07実装）

- **`streamlit_app.py`は`import app`だけでは不十分**: Streamlitはメインスクリプト
  (`streamlit_app.py`)自体は毎リラン丸ごと再実行するが、そこから`import`された
  サブモジュール(`app`)はPythonの通常のモジュールキャッシュに従うため、素の
  `import app`だと**初回リランのみ**`fase2/app.py`のトップレベルコード
  (`st.set_page_config()`・ホームページ⇔店舗トップページの切替・`app_top`/`app_a`/`app_b`への
  ディスパッチ)が実行され、2回目以降のリラン(ページ遷移やapp_a/app_bの
  ウィジェット操作)では実行されずUIが実質固まる。そのため`streamlit_app.py`は
  `app`モジュールのみ`importlib.reload`で毎リラン強制再実行する
  (`app_a`/`app_b`/`app_top`はモジュールトップレベルにst呼び出しを持たず
  関数定義のみのため、reload対象は`app`だけで足りる)。

## 制約・割り切り（再掲）

1. データ鮮度はPC(fase4)の稼働に依存（fase3は閲覧可能性のみ解決）
2. コールドスタート数十秒（無料枠の仕様）
3. 非公開アプリは1個まで（機能A/Bは`fase2/app.py`で統合済みのため1個で足りる）
4. syncとSecretsの管理対象が2DB分になる（案Cのコスト。bootstrapのコードパスは共通化する）
