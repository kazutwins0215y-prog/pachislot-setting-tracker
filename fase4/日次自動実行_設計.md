# fase4: 日次自動実行 設計書

作成: 2026-07-07（実装前レビュー用）

## 目的

fase1収集→fase2評価→fase2分析・予測を毎朝無人で直列実行し、
**来店日の8:30までに「今日の予測」をiPhoneで確認できる状態**を作る。
あわせて`prediction_log`の予測サンプルを毎日欠かさず蓄積する（評価データの蓄積が最優先動機）。

## 前提・制約

| 項目 | 内容 |
|---|---|
| 実行環境 | 自宅PC（住宅用IP必須。GitHub Actions等データセンターIPはana-slo.comのCloudflareが403ブロック） |
| 電源運用 | **夜間はシャットダウンせずスリープ**。タスクスケジューラのウェイクタイマーで自動復帰（完全シャットダウンからの自動起動は不可） |
| サイト更新 | 前日分は23:00〜翌10:00の間に更新（体感5:00〜9:00、確証なし）。**ページ一括更新**（部分更新はない前提） |
| 更新の例外 | ①数日遅れて更新される ②その日だけ永久に穴あき（それ以降は正常） |
| ユーザー要件 | 来店日は**8:30までに当日予測**が欲しい。非来店日は時刻不問 |
| 観測要件 | **運用初期はサイト更新が実際に何時に行われているか記録して知りたい** |

## 全体構成

```
タスクスケジューラ（2本）
├─ 朝タスク  6:30 起動（スリープ解除あり）
│    └─ py -3.12 fase4/run_daily.py --mode morning
│         ① ポーリングループ（6:30〜8:15、15分間隔）
│         │    fase1/メイン.py を実行（未取得日のみリクエスト＝空振りは軽い）
│         │    → 新規取得できた(店舗,日付)を collection_log.csv に検知時刻付きで追記
│         │    → 全店舗の昨日分が揃ったらループ脱出／8:15で打ち切り
│         ② fase2/evaluate_predictions.py   … 前回予測の答え合わせ
│         ③ fase2/run_store_profile.py 全店舗 … スコア再計算＋今日の予測をログ追記
│         （8:30までに完了目標）
└─ 追いタスク 10:30 起動
     └─ py -3.12 fase4/run_daily.py --mode catchup
          fase1収集1回＋②③（朝に間に合わなかった店舗の回収。
          データが進んだ店舗だけ予測が追記される＝重複ガードで自動制御）
```

### 実行順序の理由

- ②を③より先: 新しい予測を積む前に、前回予測の答え合わせを確定させる。
- ポーリング15分間隔: 更新時刻の観測分解能=15分。1回の空振りは店舗あたり1リクエスト
  （`compute_remaining_days`が未処理日だけを対象にするため）。
  6:30〜8:15で最大8回×5店舗=40リクエスト程度で礼儀の範囲内。

## run_daily.py の仕様

### 設定値（ファイル冒頭の定数。将来config化も可）

| 定数 | 初期値 | 意味 |
|---|---|---|
| `POLL_START` / `POLL_DEADLINE` | 06:30 / 08:15 | ポーリング窓（タスク起動が遅れた場合はその時刻から開始） |
| `POLL_INTERVAL_MIN` | 15 | ポーリング間隔（分）。更新時刻の傾向が掴めたら30分等へ緩和 |
| `PY_FASE1` | `py -3.12` | fase1実行用Pythonランチャ |

### 処理フロー（--mode morning）

1. レプリカから店舗ごとの`max(日付)`を取得（開始時スナップショット）
2. ループ: `fase1/メイン.py`をサブプロセス実行
   - 実行後にレプリカを再照会し、**新たに現れた(ホール名, 日付)** を
     `ホールデータ/collection_log.csv` に追記:
     `対象日, ホール名, 検知日時, ポーリング回数(その朝の何回目か), mode`
   - 全店舗が「昨日分あり」になったらループ脱出
   - 終了コード異常のうち**403（全店舗中止）を検知したら即ループ中止**
     （ブロックされたIPで叩き続けない）。403時は②③もスキップして異常ログを残し終了
3. `fase2/evaluate_predictions.py` 実行
4. `fase2/run_store_profile.py` を**引数なし（全店舗）**で実行
5. 実行サマリ（各店舗の最終データ日・予測追記件数・所要時間）を
   `fase4/logs/run_daily_YYYYMMDD.log` に出力

### --mode catchup（10:30）

ポーリングせず収集1回→②③。朝と同一コードパスで、差分は窓の有無のみ。
非来店日の運用もこれで担保（朝タスクが動かなくても1日1回はここで回る）。

### 403検知の方法

`メイン.py`はAccessForbiddenError時に全店舗中止する既存実装を利用。
run_daily側はサブプロセスの終了コード＋標準エラーの識別文字列で判定
（実装時に`メイン.py`の終了コード仕様を確認し、必要なら403時に専用終了コード
（例: exit 43）を返す小改修を入れる）。

## サイト更新時刻の観測（運用初期の要件）

- `ホールデータ/collection_log.csv`（append-only・Git管理外）に検知時刻を蓄積。
- 15分間隔ポーリングなので「検知時刻=更新時刻の上界（最大15分遅れ）」。
- 1ヶ月程度貯めたら分布を確認し、ポーリング窓・間隔・打ち切り時刻を調整する
  （例: 実際は7:00までにほぼ更新されるなら窓を6:45〜7:30へ短縮）。
- 遅延更新（例外①）はcatchup以降の検知として同じCSVに記録されるため、
  「何日遅れたか」も同時に観測できる。

## prediction_log 重複追記ガード（必須の随伴改修）

データが進んでいない店舗で③を再実行すると、append-onlyの`prediction_log`に
同じ予測が重複追記される。ガードは**`score.write_prediction_log`側**に入れる
（run_daily以外の手動実行でも効くように単一の防波堤とする）:

- 追記前に同一 `(ホール名, 予測種別, 使用データ最終日)` の行が既存なら**その店舗分をスキップ**。
- これによりrun_daily側は「進んだ店舗の判定」を持たず、常に全店舗③を回すだけの
  単純な作りにできる（catchupや手動再実行も安全）。

## fase1 側の随伴変更

| 変更 | 内容 | 理由 |
|---|---|---|
| `COLLECT_UNTIL_DAYS_AGO` 2→1 | 収集対象を「昨日まで」に短縮 | ページ一括更新前提なら中間状態の取り込みリスクなし。未更新日は`slot_data`に入らず既存の14日ギャップ再試行が拾うため安全。**これにより翌日予測の対象が「今日」になる**（現状は常に昨日=答え合わせ専用） |

例外②（永久穴あき）は現状どおり14日再試行後に自然放棄でよい（fase2は欠損日をNaN扱い）。

## タスクスケジューラ設定

| 項目 | 朝タスク | 追いタスク |
|---|---|---|
| トリガー | 毎日 6:30 | 毎日 10:30 |
| スリープ解除 | **有効**（「タスクの実行時にスリープを解除する」） | 有効 |
| 開始逃し時 | 「スケジュールされた開始時刻を逃した場合、すぐにタスクを開始する」有効 | 同左 |
| 条件 | AC電源時のみ実行は**無効**推奨（ノートPCなら要検討） | 同左 |

前提として電源オプションで「ウェイクタイマーを許可」を有効にすること。
登録は`schtasks`コマンド（またはXMLインポート）で行い、コマンドを設計書兼skillに残す。

### 実運用で判明した追加事項（2026-07-07・laptop実機）

「ノートPCなら要検討」としていたリスクが実際に発生した。`WakeToRun`はPCを
起こす（電源投入）ことは保証するが、**起動後にOSが再度スリープへ戻ることは
防がない**。実機ではバッテリー駆動時にモダンスタンバイが「Austerity Battery
Drain Budget Exceeded」「Standby Battery Budget Exceeded」等の理由で強制的に
より深いスリープ/休止へ移行し、実行中の`run_daily.py`が巻き込まれて強制終了
（exit `0xC000013A`）する事例を確認した（`Get-WinEvent -LogName System`で
電源イベントとタスク起動時刻が完全一致することで特定）。

対策として`run_daily.py`起動直後に`SetThreadExecutionState`（Win32 API、
`ES_CONTINUOUS | ES_SYSTEM_REQUIRED | ES_AWAYMODE_REQUIRED`）でOSにスリープ
防止をリクエストし、終了時（`finally`）に解除する処理を追加した。ただし
これはベストエフォートであり、**根本対策は朝6:30・10:30の時間帯にPCをAC電源へ
接続しておくこと**（バッテリー駆動由来の強制スリープ自体を回避できるため）。
詳細は[`日次自動実行_skill.md`](日次自動実行_skill.md)のトラブルシューティング参照。

## 8:30に間に合わない日の扱い（物理的限界）

サイト更新が8:15より遅い日は当日朝の予測は原理的に不可能。
その日は機能B側で「使用データ最終日」が1日古いことが分かる表示になっている
（`prediction_log`最新行の`使用データ最終日`をapp_topが表示済み）ため、
fase4側での追加対応はしない。頻度はcollection_log.csvで観測して判断。

## iPhoneからの閲覧（fase3との接続）

fase3 Stage B（クラウドホスティング）完成までは、朝の時間帯にPCが起きていることを
利用してTailscale等で自宅PCのStreamlitへアクセスする暫定運用。
fase4完了後、fase3 Stage Bで「日次ジョブ末尾に分析成果物をクラウドへpush」を追加する設計余地を残す
（run_daily.pyの⑤サマリ出力の後段にフックポイントを想定）。

## 実装物一覧・実装順

## 実装時に設計から変えた点（2026-07実装セッション時点）

- fase1呼び出し(`py -3.12`)とfase2呼び出し(`python`)でPythonランチャが異なるため、
  設計時点の単一定数`PY_FASE1`を`PY_FASE1_CMD`/`PY_FASE2_CMD`の2定数に分けた
  （fase2は既存運用どおり`python`コマンドを使う。libsql依存はfase1側に閉じているため）
- 実行サマリに`実装指示書.md`が要求する「各ステップの所要時間・終了コード」
  「collection_log追記件数」も出力するよう`RunStats`クラスで集計を追加
  （店舗ごとの最終データ日・昨日分有無に加えて表示）
- タスクスケジューラへの実際の登録(`Register-ScheduledTask`実行)はPCの自動起動設定を
  変更する持続的な操作のため、実装セッションでは登録コマンドの提示のみに留め、
  実行はユーザー自身の判断で行う運用にした（コマンドは`日次自動実行_skill.md`参照）
