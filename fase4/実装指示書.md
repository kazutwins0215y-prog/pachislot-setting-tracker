## タスク1: `fase2/score.py` — prediction_log重複追記ガード

**目的**: データが進んでいない店舗で`run_store_profile.py`を再実行しても、同じ予測が
append-onlyの`prediction_log`に二重記録されないようにする。fase4がcatchup実行や
リトライで同じ日に複数回③を回す前提のため必須。

**仕様**:
- `write_prediction_log`内で、INSERT前に既存の`(ホール名, 予測種別, 使用データ最終日)`
  の組を照会し、**既存の組に該当するrowsを除外**してから追記する。
- DELETE/UPDATEは絶対にしない(append-only維持)。全行スキップなら何もINSERTしない。
- スキップした場合はその旨をprint/loggerで出力(件数と組)。
- 粒度は台単位ではなく(ホール名, 予測種別, 使用データ最終日)単位でよい
  (1回のrun_store_profile実行=1店舗×1予測種別×1データ最終日のバッチのため)。

**検証**:
1. 任意の1店舗で`python run_store_profile.py --hole <店舗名>`を2回連続実行
2. 2回目の後、`prediction_log`の行数が増えていないこと(sqlite3で件数確認)
3. `evaluate_predictions.py`が従来どおり動くこと

## タスク2: `fase1/メイン.py` — 収集範囲短縮と403終了コード

**変更点2つ**:

1. `COLLECT_UNTIL_DAYS_AGO = 2` → `1`
   - 根拠: サイトは前日分を23:00〜翌10:00(体感5:00〜9:00)に**ページ一括**で更新するため
     中間状態の取り込みリスクがない。未更新日は`slot_data`に入らず既存の
     `RETRY_LOOKBACK_DAYS=14`ギャップ再試行が翌日以降拾うので安全。
   - 効果: 翌日予測の対象が「昨日」から「今日」になる(8:30要件の成立に必須)。

2. 403時に**専用終了コード43**で終了する
   - 現状`main()`は`AccessForbiddenError`をcatchしてログ出力後、`sync_replica`を経て
     **正常終了(exit 0)** する。呼び出し側(run_daily)が403を判別できないため、
     フラグを立てて`sync_replica`・`con.close()`は従来どおり実行した上で、
     最後に`sys.exit(43)`する形に変更。
   - 通常終了は0のまま。その他の例外は現状の挙動を変えない。

**検証**:
- 通常実行で従来どおり動き、exit code 0であること
- `AccessForbiddenError`を一時的にraiseさせる(またはモック)とexit 43になり、
  かつ`sync_replica`が呼ばれていること

## タスク3: `fase4/run_daily.py` — 日次オーケストレータ本体(新規)

**配置**: `fase4/run_daily.py`。`fase4/logs/`をログ出力先として自動作成。

**CLI**: `py -3.12 run_daily.py --mode morning|catchup`
(run_daily自体をpy -3.12で動かせばfase1呼び出しと統一できる。run_daily自体は
標準ライブラリのみで書くこと=どのPythonでも動く保険)

**冒頭の設定定数**:
```python
POLL_START = "06:30"        # これより前に起動されたら待機
POLL_DEADLINE = "08:15"     # ポーリング打ち切り時刻
POLL_INTERVAL_MIN = 15      # ポーリング間隔(分)
EXIT_CODE_FORBIDDEN = 43    # メイン.pyの403終了コード
```

パスはこのファイルからの相対で解決する(`BASE = Path(__file__).resolve().parent.parent`):
レプリカ=`BASE/'ホールデータ'/'turso_replica.db'`、収集ログCSV=`BASE/'ホールデータ'/'collection_log.csv'`、
店舗リスト=`BASE/'fase1'/'stores.json'`。

**処理フロー(--mode morning)**:

1. `fase4/logs/run_daily_YYYYMMDD.log`へのロギングを初期化(コンソールにも出力)
2. 現在時刻がPOLL_STARTより前なら待機、POLL_DEADLINE以降なら収集1回だけ実行して4へ
3. ポーリングループ(POLL_DEADLINEまで):
   a. レプリカから店舗ごとの処理済み日付集合のスナップショットを取る
      (`SELECT DISTINCT ホール名, 日付 FROM slot_data`、read-onlyで接続:
      `sqlite3.connect('file:...?mode=ro', uri=True)`)
   b. `fase1/メイン.py`をサブプロセス実行(`py -3.12`、cwd=`fase1/`)
      - exit code 43 → 「403ブロック検知・本日の処理を中止」をERRORログに残し、
        **ポーリング・評価・分析すべてスキップして exit 1**
      - その他の非0 → WARNINGログを残しループ継続(一時的失敗扱い)
   c. スナップショットとの差分=新たに現れた(ホール名, 日付)を検出し、
      `collection_log.csv`に追記(→下記フォーマット)
   d. 全店舗が「昨日(today-1)のデータあり」になったらループ脱出
   e. now+POLL_INTERVAL_MINまでsleep。POLL_DEADLINE超過なら脱出
4. `fase2/evaluate_predictions.py`をサブプロセス実行(cwd=`fase2/`)
   - 失敗しても5へ進む(答え合わせの失敗で予測追記を止めない)。ERRORログは残す
5. `fase2/run_store_profile.py`を**引数なし(全店舗)** でサブプロセス実行(cwd=`fase2/`)
   - タスク1のガードにより、データが進んでいない店舗の予測は自動スキップされる
6. サマリをログ出力: 店舗ごとの最終データ日/昨日分が揃ったか/ポーリング回数/
   collection_log追記件数/各ステップの所要時間・終了コード

**処理フロー(--mode catchup)**: 手順3のループなし版(収集1回→c.のCSV追記→4→5→6)。
POLL_START/DEADLINEは無視する。コードパスはmorningと共通化すること。

**collection_log.csv フォーマット**(append-only、ファイルが無ければヘッダー行を書く):
```
対象日,ホール名,検知日時,ポーリング回数,mode
2026-07-07,メッセ武蔵境,2026-07-08T07:15:23,3,morning
```
- 「検知日時」=run_dailyが差分を確認した時刻(サイト更新時刻の上界、分解能15分)
- 「ポーリング回数」=その実行内で何回目の収集試行で現れたか(catchupは常に1)
- 昨日分に限らず、ギャップ再試行で埋まった過去日も記録する(遅延更新の観測のため)
- **このCSVは運用初期の主目的「サイト更新が実際に何時か知る」のためのデータ。
  消さない・上書きしない**

**検証**:
1. `--mode catchup`を手動実行し、収集→評価→分析が直列に走りログとサマリが出ること
2. 続けてもう一度`--mode catchup` → run_store_profileは走るが予測追記0件
   (タスク1ガードの動作確認)、collection_log.csvに新規行なし
3. `--mode morning`をPOLL_DEADLINE後の時刻に実行 → 収集1回で4へ進むこと
4. (可能なら)POLL_INTERVAL_MINを1分・POLL_DEADLINEを近い時刻に一時変更して
   ループ・打ち切り・sleepの挙動を確認

## タスク4: タスクスケジューラ登録 + 運用skill作成

**登録する2タスク**(PowerShellの`Register-ScheduledTask`を使う。schtasks /createは
スリープ解除オプションを指定できないため):

| | 朝タスク | 追いタスク |
|---|---|---|
| 名前 | `PachislotDaily_Morning` | `PachislotDaily_Catchup` |
| トリガー | 毎日 06:30 | 毎日 10:30 |
| 操作 | `py -3.12 <絶対パス>\fase4\run_daily.py --mode morning` | 同 `--mode catchup` |
| 設定 | `-WakeToRun`(スリープ解除) + `-StartWhenAvailable`(開始逃し時に即実行) | 同左 |

```powershell
$settings = New-ScheduledTaskSettingsSet -WakeToRun -StartWhenAvailable
# ノートPCの場合は -AllowStartIfOnBatteries も検討
```

**あわせて必要な手動設定**(skillに手順として明記):
- 電源オプション「スリープ解除タイマーの許可」を有効化
- 運用前提「夜間はシャットダウンせずスリープ」(完全シャットダウンからは自動起動不可)

**`fase4/日次自動実行_skill.md`を新規作成**し、以下を記載:
- 2タスクの登録コマンド(コピペで再登録できる形)・解除コマンド
- 日々の確認ポイント(`fase4/logs/`のサマリ、collection_log.csvの見方)
- 手動で回したい場合のコマンド(`py -3.12 run_daily.py --mode catchup`)
- トラブルシューティング(exit 43=403ブロック時は日を改める、PCがオフだった日は
  StartWhenAvailableで起動後に追いつく、等)
- 運用初期の観測タスク: collection_log.csvが1ヶ月分たまったら検知時刻の分布を確認し
  POLL_START/DEADLINE/INTERVALを実態に合わせて調整する
